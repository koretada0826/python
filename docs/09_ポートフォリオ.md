# Day 9:ポートフォリオ制作 - 「お願いしたい」と言わせる作品集

## 💡 これは何?(小学生でもわかる説明)

****自分の作品集**を見せる場所**

### なぜ大事?
ランサーズで「これ作れます」を **証明できる** = 採用率 2〜3倍。
**証明がない人** vs **証明がある人**、依頼する側はどっちを選ぶ?
就活でいう **履歴書 + 作品集**。これがないと相手にされない。

### イメージ
- **ポートフォリオ** = 「**自分のアピール用パンフレット**」
- **README** = 「**作品の説明書**」(これが一番大事)
- **デモ GIF** = 「**動いてる証拠ビデオ**」(口で説明より100倍効く)
- **ストーリー(Before/After)** = 「**ビフォーアフター CM**」
- **数字** = 「**月20時間削減**」みたいな信頼の根拠
- **価格表** = 「**メニュー表**」(値段を先に見せる安心感)

### Claude Code に何を頼めばいいか
この章で覚えるのは「**何ができるか**」と「**何を頼めばいいか**」だけでOK。
具体的なコードは Claude Code に書かせる。
**README・キャッチコピー・事例ストーリーも全部 Claude に書かせる**。

## 1. ポートフォリオの正解(=「これはダメ・これは OK」)(10分)

### 1-1. NG ポートフォリオ

- コードだけ GitHub に push
- README なし
- スクショなし
- 「私のポートフォリオです」とだけ書いてある

→ **見ても価値が伝わらない**。採用されない。

### 1-2. OK ポートフォリオ

- ✅ **ストーリー**(誰のどんな悩みを解決した?)
- ✅ **数字**(時間削減◯時間/月、売上◯%UP)
- ✅ **ビジュアル**(スクショ・動画)
- ✅ **すぐ使える**(デモ URL or サンプル動作)
- ✅ **技術スタック明記**

> **発注者目線で考えると分かる:**
> 「**コードを書ける人**」より「**自分の課題を解決できる人**」を選ぶ。
> ポートフォリオは「**課題解決の実績**」を見せる場所。

## 3. 各作品の磨き方(=「磨いて売り物に」)(各40分)

### 3-1. 完成度を上げる

- [ ] **エラーハンドリング**追加(クラッシュしない)
- [ ] **サンプルデータ**自動生成スクリプト同梱
- [ ] **requirements.txt** 最新化
- [ ] **型ヒント**追加(コード品質 UP)
- [ ] **テストコード**1つでも追加(信頼度 UP)

### 3-2. README を「**売り文句**」に

各作品の README に必ず入れる5要素:

```markdown

## 解決する問題
〇〇の手作業を自動化、月△時間削減。

## デモ
![スクショ](docs/demo.gif)
👉 ライブデモ: https://...

## 主な機能
- 機能1(数字付き)
- 機能2
- 機能3

## 技術スタック
- Python 3.12
- pandas, openpyxl
- Streamlit
- Claude API

## 使い方
\`\`\`bash
pip install -r requirements.txt
python3 main.py
\`\`\`

## 想定効果(ROI)
| 項目 | 値 |
|---|---|
| 削減時間 | 月20時間 |
| 人件費換算 | 月60,000円 |
| 年間効果 | 720,000円 |

## 過去の納品実績
- EC 事業者A様(月15時間削減)

## ライセンス
MIT
```

→ クロードに「このコードを売り出す README をマークダウンで書いて」で生成可能。

> **README が一番大事な理由:**
> GitHub でリポジトリを開いた人は **README を最初に見る**。
> ここで「**価値**」が伝わらないと、コードを見るまでもなく閉じられる。

## 4. スクショ・動画(=「動く証拠」)(各30分)

### 4-1. スクショの撮り方(Mac)

| 用途 | 方法 |
|---|---|
| 一部範囲 | `Cmd + Shift + 4` → 範囲選択 |
| 画面全体 | `Cmd + Shift + 3` |
| 特定ウィンドウ | `Cmd + Shift + 4` → スペース |

### 4-2. GIF 動画(動きを見せる)

```bash
brew install --cask kap
```

**Kap** で画面録画 → GIF 出力。
または **LICEcap**(Windows / Mac)。

> **なぜ GIF?**
> README に **動く絵が貼れる**(動画は再生ボタン要るが GIF は勝手に動く)。
> 「**動いてるアプリ**」を見せられるのが圧倒的に強い。

### 4-3. 撮るべきシーン

1. **入力**(コマンド or ファイルアップロード)
2. **処理中**(進捗バー)
3. **完成**(出力結果)

→ 30秒以内の GIF にすると見やすい。

### 4-4. 配置

```
01_excel_merger/
└── docs/
    ├── screenshot_input.png
    ├── screenshot_output.png
    └── demo.gif         ← README から参照
```

## 5. ⭐ ポートフォリオサイト構築(=「自販機を看板付きに」)(120分)

> GitHub だけでなく **ちゃんとした Web サイト**を持つと信用 2倍。

### 5-1. 選択肢

| 方法 | 難易度 | 効果 |
|---|---|---|
| **GitHub Pages** | 低 | 無料、HTML 静的サイト |
| **Streamlit Cloud** | 低 | 無料、デモが直接動く |
| **Notion 公開ページ** | 超低 | 無料、ノーコード |
| **Vercel + Next.js** | 中 | プロっぽい |
| **WordPress** | 中 | カスタマイズ自由 |

### 5-2. 推奨:**Streamlit でポートフォリオサイト**

理由: **デモを直接動かせる**(他の応募者と差別化)。

### 5-3. 実装(クロードに頼む)

```
Streamlit でポートフォリオサイトを作って。

構成:
- トップページ: 自己紹介・スキル・連絡先
- サイドバーで各作品を選べる
- 各作品ページ:
  - 作品名・概要
  - スクショ・GIF
  - 技術スタック
  - 「このアプリを試す」ボタン → 実際に動かせる
  - GitHub リポジトリへのリンク
- 連絡フォーム(SendGrid 連携)

5作品分のテンプレ込み。
レスポンシブ(スマホ対応)。
ダークモード切替可能。

requirements.txt も含めて完全なコード書いて。
ファイル名は portfolio.py。
```

→ クロードがコード書く。デプロイは Streamlit Cloud で無料。

### 5-4. 公開 URL

`https://<ユーザー名>-portfolio.streamlit.app/`

→ ランサーズ提案文に貼る。

## 12. よくある問題と対処

| 問題 | 原因 | 直し方 |
|---|---|---|
| Streamlit Cloud にデプロイできない | requirements.txt 未配置 | プロジェクトルートに置く |
| GIF が大きすぎて GitHub に上がらない | 100MB 超 | Kap で品質を下げる、長さを 30 秒以内に |
| README の画像が表示されない | 相対パス間違い | `docs/demo.gif` など正しいパス |
| 連絡フォームが動かない | SendGrid 未認証 | Sender Authentication 済ませる |
| アクセスしても画面が真っ白 | コンソールにエラー | ブラウザの開発者ツール(F12)で確認 |
| ポートフォリオを見ても何ができるか分からない | 価値の言語化不足 | Before/After + 数字を入れる |
| 採用されない | プロフィール顔写真がない / 文章硬すぎ | プロフィール写真 + 親しみやすい挨拶を入れる |
