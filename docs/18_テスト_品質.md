# Day 18:テスト・品質 - プロの最低条件

## 💡 これは何?(小学生でもわかる説明)

**コードが「ちゃんと動くか」を自動で味見してくれるロボット(pytest)を雇う日**

### なぜ大事?
**テスト書ける人 = 信頼される = 月額契約取りやすい**。
バグだらけのコードを納品 → 評価★1 → 次の仕事ゼロ。
「動きました!」じゃなく「**バグがないと証明できます**」が月100万の鍵。

### イメージ
- **pytest**=「**自動味見ロボット**」(料理が美味しいか何回も確認)
- **fixture**=「**料理の下ごしらえ**」(複数のテストで使い回す)
- **mock**=「**ニセモノの食材**」(本物のAPI叩かずテスト)
- **型ヒント**=「**料理の材料表**」(誰が見ても何が入るか分かる)
- **logging**=「**料理日記**」(後から「どこで失敗?」を追える)
- **CI**=「**保健所の自動検査**」(push する度に勝手に検査)

### Claude Code に何を頼めばいいか
テストコード・型ヒント追加・Linter設定・ロギング実装は全部クロード。
あなたは「**何をテストすべきか**」を考えるだけ。

## 1. なぜテストを書く?(=信用を守る盾)(15分)

### 1-1. 趣味コード vs 仕事コード

| 種類 | 求められる品質 |
|---|---|
| 趣味コード | 1度動けばOK、バグ出たら直す |
| 仕事コード | **6ヶ月後の自分や他人**が触る、追加機能で**既存が壊れない**保証が必要 |

> **「テストがない = 信頼できない」**とプロは見る。

### 1-2. テストないコードの末路

```
お客様: "新機能追加して"
あなた: "はい"
→ 既存のA機能が壊れる
→ 客が怒る
→ 信用ゼロ → 月額契約解約
```

→ **テストはあなたの信用を守る盾**。

> **イメージ:**
> 料理人が新メニュー作って**味見しないで客に出す**=危険。
> テスト=自動味見ロボット。バグを客に出す前に弾く。

## 2. pytest 入門(=味見ロボットを雇う)(40分)

### 2-1. インストール

```bash
pip install pytest
```

### 2-2. 一番簡単なテスト

`utils.py`:
```python
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("0で割れない")
    return a / b
```

`test_utils.py`(ファイル名 `test_` で始める):
```python
from utils import add, divide
import pytest

def test_add():
    # assert は「これは真であるべき」と主張
    assert add(2, 3) == 5
    assert add(-1, 1) == 0

def test_divide():
    assert divide(10, 2) == 5

def test_divide_by_zero():
    # ValueError が出ることを期待するテスト
    with pytest.raises(ValueError):
        divide(10, 0)
```

### 2-3. 実行

```bash
pytest
# or
pytest -v  # 詳細(どのテストがpassedしたか1行ずつ表示)
pytest test_utils.py::test_add  # 特定だけ
```

→ 緑の `passed` が出れば成功。

### よくある間違い

| ミス | エラー | 直し方 |
|---|---|---|
| ファイル名が `utils_test.py` | テスト発見されない | `test_` で始める(`test_utils.py`) |
| 関数名が `add_test` | テスト発見されない | `test_` で始める(`test_add`) |
| `import` が間違う | ModuleNotFoundError | `__init__.py` 作る or `pytest.ini` で path 指定 |

## 3. fixture - 共通の前準備(=料理の下ごしらえ)(30分)

### 3-1. なぜ?

> **複数のテストで同じデータを使いたい時**。
> 毎回同じセットアップを書くのは無駄 → fixture で1回書いて使い回し。

```python
import pytest

@pytest.fixture
def sample_data():
    return [1, 2, 3, 4, 5]

def test_sum(sample_data):  # 引数に fixture 名を書くと自動で渡される
    assert sum(sample_data) == 15

def test_max(sample_data):
    assert max(sample_data) == 5
```

→ `sample_data` を**両方のテストで使い回せる**。

### 3-2. よくある fixture

```python
@pytest.fixture
def db_connection():
    """DB接続を準備、テスト後にクローズ"""
    conn = sqlite3.connect(":memory:")  # メモリ上の使い捨てDB
    yield conn  # テストにこれを渡す
    conn.close()  # テスト終了後に自動で閉じる

@pytest.fixture
def temp_csv(tmp_path):
    """一時的なCSVファイル(tmp_path はpytest標準の一時フォルダ)"""
    f = tmp_path / "test.csv"
    f.write_text("a,b,c\n1,2,3")
    return f
```

> **`yield` のなぜ:**
> - `yield` 前 = テスト前のセットアップ
> - `yield` 後 = テスト後のクリーンアップ
> 例外が起きても確実にクリーンアップされる。

## 4. パラメータ化テスト(=複数パターンを一気に)(15分)

```python
@pytest.mark.parametrize("a, b, expected", [
    (2, 3, 5),
    (-1, 1, 0),
    (0, 0, 0),
    (100, 200, 300),
])
def test_add(a, b, expected):
    assert add(a, b) == expected
```

→ **1関数で4ケース**を一気にテスト。

> **メリット:** 4倍のテストが書ける + 失敗時にどのパターンで落ちたか分かる。

## 5. モック - 外部依存を切り離す(=ニセモノの食材)(30分)

### 5-1. なぜモック?

> **本物のAPIを叩くと:**
> - 遅い(1秒×100テスト = 100秒)
> - お金がかかる(OpenAI API)
> - ネットがないと動かない
>
> **モック使うと:**
> - 一瞬(1ms)
> - 無料
> - オフラインOK

```python
from unittest.mock import patch, MagicMock

def fetch_weather(city):
    import requests
    return requests.get(f"https://api.com/{city}").json()

# テスト(本物のAPIを呼ばない)
@patch("requests.get")
def test_fetch_weather(mock_get):
    # 偽のレスポンスを設定
    mock_get.return_value.json.return_value = {"temp": 25}
    result = fetch_weather("Tokyo")
    assert result["temp"] == 25
```

→ **外部APIなしでテストできる**(高速・確実)。

## 7. 型ヒント(mypy)(=料理の材料表)(30分)

### 7-1. なぜ?

Python は動的型付けだが、**型を明示**するとバグ減る。

> **型ヒントなし:**
> ```python
> def add(a, b):
>     return a + b
> ```
> → 文字列が来るのか整数が来るのか分からない
>
> **型ヒントあり:**
> ```python
> def add(a: int, b: int) -> int:
>     return a + b
> ```
> → IDE が「整数しかダメ」と教えてくれる

### 7-2. 基本

```python
def add(a: int, b: int) -> int:
    return a + b

def fetch_user(id: int) -> dict[str, str]:
    return {"name": "太郎", "email": "x@y.z"}

# Optional(値があるかも)
from typing import Optional
def find_user(id: int) -> Optional[dict]:
    if id == 1: return {"name": "太郎"}
    return None
```

### 7-3. 型チェックツール

```bash
pip install mypy
mypy myproject/
```

→ 型違いを実行前に検出。

### 7-4. 型ヒントのプロ作法

```python
from typing import Any, Callable, Iterator

# 関数を引数に
def apply(f: Callable[[int], int], x: int) -> int:
    return f(x)

# ジェネレータ
def count(n: int) -> Iterator[int]:
    for i in range(n):
        yield i

# 厳密にしたくない時
def process(data: Any) -> Any:
    ...
```

→ 法人案件は**型ヒントが標準**。書けないと信頼下がる。

```toml
[tool.ruff]

ignore = ["E501"]
```

### 8-3. VS Code 連携

`.vscode/settings.json`:
```json
{
    "[python]": {
        "editor.defaultFormatter": "charliermarsh.ruff",
        "editor.formatOnSave": true,
        "editor.codeActionsOnSave": {
            "source.organizeImports.ruff": "explicit"

}
```

→ **保存するたび自動整形**。

> **なぜ重要?**
> インデントズレ・import順番ぐちゃぐちゃ・行が長すぎる…全部Ruffで自動修正。
> プロは「**保存ボタン押すと綺麗になる**」設定が当たり前。

## 9. ロギング(=料理日記)(20分)

### 9-1. `print()` は捨てる

| ❌ print | ✅ logging |
|---|---|
| 本番で消えない | レベル別に出し分け |
| ログレベルなし | DEBUG/INFO/WARNING/ERROR/CRITICAL |
| ログファイルに残らない | ファイル + コンソール両方OK |
| 構造化できない | JSON で集計可 |

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(name)s: %(message)s',
    handlers=[
        logging.FileHandler('app.log'),
        logging.StreamHandler()
    ]
)

logger = logging.getLogger(__name__)

logger.debug("詳細")    # 開発時のみ
logger.info("通常")     # 普通の動作
logger.warning("警告")   # 注意
logger.error("エラー")   # エラー(処理は継続)
logger.critical("致命")  # 致命的(停止級)
```

### 9-2. 構造化ログ(JSONL)

```python
import json
logger.info(json.dumps({
    "event": "user_login",
    "user_id": 123,
    "ip": "1.2.3.4"
}))
```

→ 後から検索・集計しやすい。

## 11. CI でテスト自動化(=自動検査)(15分)

`.github/workflows/test.yml`:
```yaml
name: Test
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install -r requirements.txt
      - run: ruff check .
      - run: mypy .
      - run: pytest --cov=.
```

→ push する度に Lint + 型チェック + テスト 自動実行。
