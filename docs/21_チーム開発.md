## 2. コードレビュー(30分)

### 受ける側(自分のPRが見られる)

#### 良い PR

```markdown
## 概要
ユーザー検索機能を追加しました。

## 変更内容
- `GET /users?q=keyword` エンドポイント追加
- 部分一致(LIKE)検索

## テスト
- [x] 単体テスト 5件追加
- [x] ローカルで動作確認
- [x] CI 緑

## 設計判断
全件SELECTせず、SQLでフィルタしてます。
1万件超のテーブル想定で、IndexはまだなしのままにしてOKでしょうか?

## スクショ
(実行結果)

Closes #42
```

→ **背景・判断を書く**とレビューが速い。

#### 指摘されたら

- ✅ 素直に受け入れる
- ✅ 反論する時は**理由+代案**
- ✅ 修正後は「修正しました」と返信
- ❌ 感情的・無視・遅延

#### 例:良い反論

```
ご指摘ありがとうございます。
ループでなく内包表記にすべき、とのことですが、
今回はデバッグ性を優先してループのままにしたいです。
理由: 1要素ごとに print デバッグ可能なので。
代案: ループの中の処理を関数化して内包表記に直せばどうでしょう?
```

→ **建設的議論**。

### する側(他人のPRを見る)

#### 確認順

1. **PR の説明**を読む(意図理解)
2. **変更ファイル一覧**(影響範囲)
3. **テストがあるか**(なければ要求)
4. **動作確認**(checkout して試す)
5. **コード読む**(命名・構造・効率)

#### 良いコメント

```
✅ "[questions] この場合 None が返るとどうなる?"
✅ "[suggest] 30行超えてるので関数化どうでしょう"
✅ "[nit] 細かいですが命名 `data` より `users` の方が...?"
✅ "[blocker] ここSQLi の可能性あり、プレースホルダ必須です"
✅ "[praise] このリファクタきれい!"
```

→ **タグで重要度**示す。`[nit]`(些細) `[blocker]`(必須修正) など。

#### NGコメント

```
❌ "ダメ" (理由なし)
❌ "もっとちゃんと書いて"
❌ 個人攻撃
❌ 大量の些細指摘で疲弊させる
```

## 3. 質問の仕方(15分)

### 悪い質問


```

## 仮説
- 列名が違う?

## 質問
列名は確かに `name` のはずだが、何が原因と考えられますか?
```

## 7. リファクタリング(20分)

「**動きを変えずにコードを綺麗にする**」。

### 三大原則

1. **DRY**(Don't Repeat Yourself)同じコード書かない
2. **KISS**(Keep It Simple, Stupid)単純が一番
3. **YAGNI**(You Aren't Gonna Need It)未来の機能を先に作らない

### よくあるリファクタ

#### Before
```python
def get_total(items):
    total = 0
    for item in items:
        if item['price'] > 0:
            if item['stock'] > 0:
                total += item['price']
    return total
```

#### After
```python
def get_total(items):
    return sum(
        item['price']
        for item in items
        if item['price'] > 0 and item['stock'] > 0
    )
```

→ **可読性UP**、テスト変更不要。

### Extract Function

長い関数を**意味のある単位**に分割。

```python
# Before(50行の関数)
def process():
    # データ取得
    ...20行
    # 検証
    ...15行
    # 保存
    ...15行

# After
def process():
    data = fetch_data()
    validate(data)
    save(data)
```

## 8. 設計パターン(基本だけ)(20分)

### Singleton(単一インスタンス)

```python
class DatabaseConnection:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance
```

### Factory(生成を関数化)

```python
def create_storage(type: str):
    if type == "s3": return S3Storage()
    if type == "local": return LocalStorage()
    raise ValueError(f"Unknown: {type}")
```

### Strategy(振る舞い差し替え)

```python
class PaymentProcessor:
    def __init__(self, gateway):
        self.gateway = gateway  # Stripe / PayPal / Square
    def charge(self, amount):
        return self.gateway.process(amount)
```

→ **これだけ知ってれば設計レビュー突破**。

## 9. 経験して学ぶ・自分1人でも擬似的に(15分)

### 1人でもできる擬似チーム開発

1. **GitHubのリポジトリで自分にPR出す**
2. **時間置いて自分でレビュー**(他人として読む)
3. **コミットを意味のある単位で分ける**
4. **CHANGELOG・README を毎日更新**
5. **Issue 駆動で開発**(タスクをIssueに書く)

→ **これだけで「チーム開発経験」**になる。

### オープンソース貢献(中級向け)

1. 興味あるリポジトリ選ぶ
2. Issue を見て簡単なやつから
3. PR 出す
4. レビュー受ける → 修正
5. マージされる

→ **GitHub の貢献履歴**が経歴の証拠。
