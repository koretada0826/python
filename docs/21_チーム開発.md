# Day 21:チーム開発 - SES・法人案件で求められる作法

## 💡 これは何?(小学生でもわかる説明)

**他人と一緒に大きいシステムを作る作法**

### なぜ大事?
1人で作るプログラムは、せいぜい「自分の家の小屋」くらい。
でも、お給料が高い仕事(SES案件・大手企業)は、**100人で作る大豪邸**みたいなもの。
1人作法のままだと、釘の打ち方も、木材の渡し方もバラバラで、家が崩れる。

**チーム開発作法ができる人 = 単価が一気に上がる**。
逆に「コードは書けるけどチームで動けない人」は単価60万円が天井で止まる。

### イメージ
- **1人開発**=「砂場でひとりでお城を作る」
- **チーム開発**=「みんなで巨大なレゴ城を作る」(誰がどのブロックを担当するか、設計図はどこにあるか、ぶつからずに作るルールが必要)
- **アジャイル**=「1階作って住んでみて、ダメなら直す」をくり返す家づくり
- **コードレビュー**=「料理を出す前に他の料理人が味見してくれる」
- **Pull Request (PR)**=「この料理をお店に出していいですか?」とお伺いを立てる紙

### Claude Code に何を頼めばいいか
この章で覚えるのは「**何ができるか**」と「**何を頼めばいいか**」だけでOK。
具体的なコードや文章は Claude Code に書かせる。

## 💰 この章後の月収:**+10〜30万円**(SES高単価)
## ⏱ 所要:2-3時間

「**1人開発しかしたことない**」エンジニアは月60万円が天井。
チーム開発作法を身につけると**月80-150万円**。

> 大手企業は「コードが書ける人」ではなく「**チームに混ぜても事故を起こさない人**」を欲しがる。
> 事故を起こさない人=高単価。コードが書けるだけ=低単価。

## 1. アジャイル/スクラムって何?(20分)

### 1-1. ウォーターフォール vs アジャイル

> **ウォーターフォール**=「**完成図を全部書いてから家を建てる**」(=昔ながら)
> **アジャイル**=「**1階作って住んでみて、ダメなら直す**」(=今主流)

| 古い(WF) | 新しい(アジャイル) |
|---|---|
| 全部設計してから作る | 小さく作って改善を繰り返す |
| 半年〜1年単位 | 1-2週間単位 |
| 仕様変更しづらい | 変更前提 |
| 客の声が後で反映される | 客の声がすぐ反映される |

→ **法人開発の80%はアジャイル**。

> **なぜアジャイル主流?**
> 客は「使ってみるまで本当に欲しいものを知らない」から。
> 1年かけて作ったら「これじゃない」と言われたらやり直し。1週間ごとに見せれば、ズレを早く直せる。

### 1-2. スクラム用語(=アジャイルのやり方の1つ)

スクラムは「**ラグビーのスクラム**」が語源。みんなで肩組んでボールを前に進めるイメージ。

| 用語 | 意味 | たとえ話 |
|---|---|---|
| **スプリント** | 1-2週間の開発サイクル | マラソンの「1区間」 |
| **プロダクトバックログ** | やることリスト | レストランの注文リスト |
| **デイリースクラム** | 毎朝15分の状況共有 | 部活の朝練の点呼 |
| **レトロスペクティブ** | 振り返り会(週末) | 試合のあとの反省会 |
| **ベロシティ** | 1スプリントでこなせる量 | 1日に走れるキロ数 |
| **ストーリーポイント** | タスクの大きさ(数字) | 山登りの難易度(★1〜5) |
| **プロダクトオーナー(PO)** | 「何を作るか」決める人 | レストランのオーナー |
| **スクラムマスター** | チームを支える進行役 | 部活の顧問 |

### 1-3. ジュニアの動き方

- **朝会で「昨日やったこと/今日やること/困りごと」を1分で**(長くしない)
- **タスクは小さく分解**(1日で終わるサイズ。3日かかるタスクは分割し直す)
- **ハマったら30分で相談**(自分で抱え込まない=チームの時間を奪う行為)

> **なぜ30分?**
> 1時間ハマる=チームの貴重な作業時間1時間がムダ。
> 5分の質問で5分で解決するなら、それが正解。「自力で解こう」は美徳ではない。

## 2. コードレビュー(=他人にコードを見てもらう)(30分)

> **イメージ:** 料理を客に出す前に、別の料理人が味見してくれる。
> 「塩入れ忘れてるよ」「肉が生焼けだよ」と教えてくれる。
> これがないと客(=本番環境)に生焼け料理を出してしまう。

### 2-1. PR(Pull Request)って何?

「**この料理(=コード変更)、お店(=メインブランチ)に出していいですか?**」とお伺いを立てる紙。

```
ローカル(自分のPC)
  ↓ コード書く
GitHub の自分のブランチ(個室)
  ↓ PR 作成
レビュアーがチェック
  ↓ OK
main ブランチ(本番に近い場所)に取り込まれる(マージ)
```

### 2-2. 受ける側(自分のPRが見られる)

#### 良い PR の書き方

```markdown
## 概要
ユーザー検索機能を追加しました。

## 変更内容
- `GET /users?q=keyword` エンドポイント追加
- 部分一致(LIKE)検索

## テスト
- [x] 単体テスト 5件追加
- [x] ローカルで動作確認
- [x] CI 緑

## 設計判断
全件SELECTせず、SQLでフィルタしてます。
1万件超のテーブル想定で、IndexはまだなしのままにしてOKでしょうか?

## スクショ
(実行結果)

Closes #42
```

→ **背景・判断を書く**とレビューが速い。

> **なぜ「設計判断」を書く?**
> レビュアーは「なぜこう書いたか」が分からないと、的外れな指摘をしてしまう。
> 先に書いておけば、議論が建設的になる。

#### 指摘されたら

| 反応 | OK/NG | 理由 |
|---|---|---|
| 素直に受け入れる | ✅ | 学びになる |
| 反論する時は理由+代案 | ✅ | 建設的 |
| 修正後「修正しました」と返信 | ✅ | レビュアーが再確認しやすい |
| 感情的になる | ❌ | チームの空気が悪くなる |
| 無視・遅延 | ❌ | プロジェクトが止まる |

#### 例:良い反論

```
ご指摘ありがとうございます。
ループでなく内包表記にすべき、とのことですが、
今回はデバッグ性を優先してループのままにしたいです。
理由: 1要素ごとに print デバッグ可能なので。
代案: ループの中の処理を関数化して内包表記に直せばどうでしょう?
```

→ **建設的議論**。「なぜそうしたか」+「じゃあこうしませんか?」のセットがプロ。

### 2-3. する側(他人のPRを見る)

#### 確認順

1. **PR の説明**を読む(意図理解)
2. **変更ファイル一覧**(影響範囲)
3. **テストがあるか**(なければ要求)
4. **動作確認**(checkout して試す)
5. **コード読む**(命名・構造・効率)

#### 良いコメント(タグ付き)

```
✅ "[questions] この場合 None が返るとどうなる?"
✅ "[suggest] 30行超えてるので関数化どうでしょう"
✅ "[nit] 細かいですが命名 `data` より `users` の方が...?"
✅ "[blocker] ここSQLi の可能性あり、プレースホルダ必須です"
✅ "[praise] このリファクタきれい!"
```

→ **タグで重要度**示す。

| タグ | 意味 | 対応 |
|---|---|---|
| `[blocker]` | 必ず直して | マージ前必須 |
| `[suggest]` | 提案 | 議論してOK |
| `[nit]` | 些細(nitpick) | 直さなくてOK |
| `[questions]` | 質問 | 答えるだけ |
| `[praise]` | ほめ言葉 | レビュー受ける側のモチベUP |

#### NGコメント

```
❌ "ダメ" (理由なし)
❌ "もっとちゃんと書いて"
❌ 個人攻撃
❌ 大量の些細指摘で疲弊させる
```

> **覚えておく:**
> レビューは「**コードを批判する場**」であって「人を批判する場」ではない。
> 「あなたが」ではなく「このコードが」と書く。

## 3. 質問の仕方(15分)

> **これは超大事**。質問のうまさで「使える人」かどうか判定される。

### 3-1. 悪い質問

```
"動きません"
```

→ レビュアーが10往復しないと解決しない。**チームの時間を奪う罪**。

### 3-2. 良い質問テンプレ(=コピペ用)

```
## やりたいこと
ユーザー検索を SQLAlchemy で実装したい

## やってみたこと
1. `db.query(User).filter(User.name == name).all()` を書いた
2. 部分一致したいので `like()` を試した

## エラー
sqlalchemy.exc.OperationalError: no such column: User.name

## 仮説
- 列名が違う?
- セッションの問題?

## 質問
列名は確かに `name` のはずだが、何が原因と考えられますか?
```

→ **30分かけてこの質問にすると、5分で解決**。

> **なぜ仮説を書く?**
> 仮説を書くと「自分で考えた跡」が見える。
> 丸投げ質問は「この人、自分で考えてないな」と思われて評価が下がる。

### 3-3. 「ラバーダッキング」(=お風呂のあひる)

質問を文章で書いてる途中に、自分で答えに気づくことが多い。

## 7. リファクタリング(=コード磨き)(20分)

「**動きを変えずにコードを綺麗にする**」。

> **イメージ:** 部屋の家具の位置を変える。住む人(=ユーザー)から見て生活は変わらないが、住み心地(=メンテしやすさ)が良くなる。

### 7-1. 三大原則

| 原則 | 意味 | たとえ話 |
|---|---|---|
| **DRY** (Don't Repeat Yourself) | 同じコード書かない | 同じレシピを3冊にコピーしない |
| **KISS** (Keep It Simple, Stupid) | 単純が一番 | 包丁で切れるのに高級カッター使わない |
| **YAGNI** (You Aren't Gonna Need It) | 未来の機能を先に作らない | まだ来てない客のために席を100席増やさない |

### 7-2. よくあるリファクタ

#### Before

```python
def get_total(items):
    total = 0
    for item in items:
        if item['price'] > 0:           # 価格が正のもの
            if item['stock'] > 0:       # 在庫があるもの
                total += item['price']  # 合計に足す
    return total
```

#### After

```python
def get_total(items):
    # 価格が正かつ在庫があるものだけ合計
    return sum(
        item['price']
        for item in items
        if item['price'] > 0 and item['stock'] > 0
    )
```

→ **可読性UP**、テスト変更不要。

### 7-3. Extract Function(=長い関数を切り出す)

長い関数を**意味のある単位**に分割。

```python
# Before(50行の関数=何やってるか分からない)
def process():
    # データ取得
    ...20行
    # 検証
    ...15行
    # 保存
    ...15行

# After(目次みたいに読める)
def process():
    data = fetch_data()   # データ取得
    validate(data)        # 検証
    save(data)            # 保存
```

> **なぜ良い?**
> 上のコードは「目次」みたいに読める。詳細を知りたい時だけ各関数を見ればいい。

## 8. 設計パターン(基本だけ)(20分)

> **デザインパターン**=「**よくある問題のテンプレ集**」。
> 料理でいう「定番レシピ」。先人が編み出した解法。

### 8-1. Singleton(単一インスタンス)

「**世界に1つしかない**」ことを保証する。

```python
class DatabaseConnection:
    _instance = None  # クラス変数(全員で共有)

    def __new__(cls):
        # まだ作られてなければ作る、あればそれを返す
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance
```

> **なぜ?**
> DB接続が複数あると、コネクション枯渇する。1つに絞る。

### 8-2. Factory(生成を関数化)

「**何を作るか」を関数に任せる**。

```python
def create_storage(type: str):
    # 引数によって違うものを返す
    if type == "s3": return S3Storage()
    if type == "local": return LocalStorage()
    raise ValueError(f"Unknown: {type}")
```

> **なぜ?**
> 呼び出す側は「S3 か local か」を意識せずに済む。設定で切り替えられる。

### 8-3. Strategy(振る舞い差し替え)

「**やり方を後から差し替えられる**」。

```python
class PaymentProcessor:
    def __init__(self, gateway):
        self.gateway = gateway  # Stripe / PayPal / Square

    def charge(self, amount):
        return self.gateway.process(amount)
```

→ **これだけ知ってれば設計レビュー突破**。

> **なぜ?**
> 「Stripeでやる」「PayPalでやる」を if 文で書くと、決済方法追加のたびに本体改修。
> Strategy なら**新クラス追加だけ**で済む。

## 10. リモート開発のコツ(15分)

### 10-1. 必須ツール

| ツール | 用途 |
|---|---|
| **Slack/Discord** | 文章コミュニケーション |

| **Notion** | ドキュメント・タスク |

### 10-2. コミュニケーション原則

- **非同期前提**(返信は数時間後でOK)

> **なぜ非同期?**
> リモートは時差・他タスクで「即返信」できない。即返信を期待すると、お互いストレス。

### 10-3. タイムゾーン違い

```
@A さん(JST 18:00)
> 〇〇についてご相談
A さん(EST 5:00)— 朝起きたら見る
```

→ **遅延を前提**で設計。重要な質問は「いつまでに返答ほしいか」明記。

```
└── CHANGELOG.md(各日更新)
```

→ **これでSES面談突破レベル**。

## 12. よくある間違い・エラー(10分)

| 状況 | 間違いの原因 | 直し方 |
|---|---|---|
| PRが大きすぎてレビュー来ない | 機能を1PRに詰め込みすぎ | 1PR=1機能。500行以下が目安 |
| 「動きません」で質問 | 情報不足 | 「やりたいこと/試したこと/エラー」テンプレ使う |
| レビュー指摘に感情的に反応 | 「人」を批判されたと誤解 | コードへの指摘=自分への攻撃ではない |
| ハマって2時間放置 | 自力で解こうとした | 30分で相談ルール |
| READMEに書いてないことばかり | 書く文化がない | 毎PRでドキュメント更新を習慣化 |
| マージ前のコンフリクト放置 | rebaseが面倒 | 毎日 `git pull origin main` |
| ブランチ名がバラバラ | 命名規則がない | `feature/xxx`, `fix/xxx` などで統一 |

## 13. 今日学んだ用語まとめ

| 用語 | 一言で |
|---|---|
| アジャイル | 小さく作って改善するやり方 |
| スクラム | アジャイルの代表的な進め方 |
| スプリント | 1-2週間の開発単位 |
| バックログ | やることリスト |
| デイリースクラム | 朝会(15分) |
| レトロスペクティブ | 振り返り会 |
| ベロシティ | チームの開発速度 |
| PR (Pull Request) | コード取り込み依頼 |
| コードレビュー | コードを他人にチェックしてもらうこと |
| `[blocker]` | 必ず直すべき指摘 |
| `[nit]` | 些細な指摘 |
| ラバーダッキング | あひるに話す=自分で気づく |
| DRY | 同じコード書かない |
| KISS | 単純に書く |
| YAGNI | 先回りして作らない |
| リファクタリング | 動作変えずにコード磨き |
| Singleton | 世界に1つだけ |
| Factory | 生成を関数化 |
| Strategy | やり方を差し替え可能に |
| SemVer | バージョン番号の付け方 |
| ペアプロ | 2人で1画面 |
| モブプロ | 3-5人で1画面 |
