## 1. 並行処理の3種類(15分)

| 種類 | 用途 | 実体 | たとえ話 |
|---|---|---|---|
| **マルチスレッド** | I/Oバウンド(ファイル読書) | `threading` | 1人で両手作業 |
| **マルチプロセス** | CPUバウンド(画像処理) | `multiprocessing` | 分身の術で複数の自分 |
| **非同期(コルーチン)** | ネットワークI/O | `asyncio` | ウェイター並列接客 |

→ Python の GIL 制約で **CPU処理にはマルチプロセス**、**I/O処理には非同期**が王道。

> **GIL とは?**
> Global Interpreter Lock。
> 「**Pythonは同時に1スレッドしかコードを動かせない**」という制約。
> なのでスレッド並列は**CPU処理を速くしない**。I/O待ち中だけ別スレッドが動ける。

### 使い分け表

| やりたいこと | 選ぶ手段 |
|---|---|
| 1000個のAPIを叩く | asyncio + aiohttp |
| 大量画像をリサイズ | ProcessPoolExecutor |
| ファイル100個を並行で開く | ThreadPoolExecutor |
| 動画変換(超重い) | multiprocessing |
| Webアプリのバックグラウンド処理 | Celery + Redis |

## 2. asyncio 入門(40分)

### 2-1. async / await って何?

> **イメージ:** 「ラーメン屋でお湯沸かしてる間に、別の注文を取る」やり方。
> 普通=お湯が沸くまで何もしない(=直列、無駄)
> async=お湯沸かし中に他の作業をする(=並行、効率的)

```python
import asyncio

async def fetch(url: str) -> str:
    """非同期関数(async def で定義)"""
    print(f"Start: {url}")
    await asyncio.sleep(1)  # I/O待機を擬似(=1秒待つ)
    print(f"End: {url}")
    return f"Result of {url}"

async def main():
    # 順次(直列)= 約2秒
    r1 = await fetch("a")  # a 終わるまで待つ
    r2 = await fetch("b")  # b を始める

    # 並行 = 約1秒(同時に走らせる)
    results = await asyncio.gather(
        fetch("a"),
        fetch("b"),
        fetch("c"),
    )
    print(results)

asyncio.run(main())  # イベントループを起動
```

### 2-2. キーワード

| キーワード | 意味 |
|---|---|
| `async def` | コルーチン関数の定義 |
| `await` | 非同期処理の完了を待つ |
| `asyncio.gather` | 複数を並行実行 |
| `asyncio.run` | イベントループ起動 |
| イベントループ | 非同期タスクの司令塔 |

### 2-3. なぜ速い?

> 普通のコード:
> ```
> Aの結果待ち [____1秒____]
>                            Bの結果待ち [____1秒____] = 計2秒
> ```
>
> async コード:
> ```
> Aの結果待ち [____1秒____]
> Bの結果待ち [____1秒____] = 計1秒(同時)
> ```

## 3. aiohttp で 1000並行 HTTPリクエスト(30分)

### 3-1. インストール

```bash
pip install aiohttp
```

### 3-2. コード例

```python
import asyncio
import aiohttp

async def fetch_url(session, url):
    """1つのURLを取得"""
    async with session.get(url) as res:
        return await res.text()

async def main():
    urls = [f"https://api.github.com/repos/python/cpython/issues/{i}" for i in range(1, 51)]

    # セマフォで同時実行数制限(=サーバー負荷を抑える)
    sem = asyncio.Semaphore(10)  # 同時10接続まで

    async with aiohttp.ClientSession() as session:
        async def bounded(url):
            async with sem:  # セマフォを取得(=空きがあれば進む)
                return await fetch_url(session, url)

        # 50URLを同時最大10並列で取得
        results = await asyncio.gather(*[bounded(u) for u in urls])

    print(f"取得完了: {len(results)}件")

asyncio.run(main())
```

→ 普通の requests なら30秒、これなら**3秒**。

> **セマフォとは?**
> 「**入場券N枚**」のイメージ。同時にN人までしか入れない。
> 同時1000接続するとサーバーから怒られる(=BAN)ので制限する。

## 4. ThreadPoolExecutor(従来型、簡単)(20分)

```python
from concurrent.futures import ThreadPoolExecutor
import requests

def fetch(url):
    return requests.get(url).status_code

urls = ["https://example.com"] * 50

# 同時10スレッドで処理
with ThreadPoolExecutor(max_workers=10) as ex:
    results = list(ex.map(fetch, urls))

print(f"成功: {sum(1 for r in results if r == 200)}")
```

→ 既存コードを最小限の変更で並行化。

> **asyncio との違い:**
> - asyncio = 新しい・速い・コードを async に書き直す必要
> - ThreadPool = 古いコードのまま並行化できる(楽)

## 5. ProcessPoolExecutor(CPU処理)(15分)

```python
from concurrent.futures import ProcessPoolExecutor

def heavy_calc(n):
    """重いCPU計算"""
    return sum(i*i for i in range(n))

# 4プロセス並行(=4コア使い切る)
with ProcessPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(heavy_calc, [10**6] * 8))
```

→ 4コア並行で**4倍速**(CPU処理)。

> **Thread と Process の違い:**
> - Thread = 1つのプロセス内で並行(GIL制約あり)
> - Process = 分身の術で複数プロセス(GILなし、メモリ別)

## 6. Celery + Redis - バックグラウンドタスク(40分)

「**重い処理をWeb APIから切り離す**」。SaaS の必修。

> **イメージ:** レストランで「メール送信して」と言われた時、
> シェフが手を止めて10分メール送る → 客全員待たされる(=ダメ)
> 「メール係に渡しといた、続き作るね」と言ってすぐ料理に戻る(=Celery)

### 6-1. インストール

```bash
pip install celery redis
brew install redis
brew services start redis
```

### 6-2. 構成

```
[FastAPI] → タスク登録 → [Redis Queue] → [Celery Worker(別プロセス)]
              ↑                                    ↓
              └────────── 結果取得 ──────────────┘
```

### 6-3. 実装

`tasks.py`:

```python
from celery import Celery

# Redis を仲介役(broker)として使う
app = Celery("tasks", broker="redis://localhost:6379/0", backend="redis://localhost:6379/1")

@app.task
def send_email(to: str, body: str):
    # 重い処理(数十秒)
    print(f"Sending to {to}")
    return f"Sent to {to}"

@app.task
def process_csv(csv_path: str):
    # 大量データ処理
    return "done"
```

`main.py`(FastAPI):

```python
from fastapi import FastAPI
from tasks import send_email

app = FastAPI()

@app.post("/notify")
def notify(email: str):
    # .delay() で即座にreturn(=処理は裏で走る)
    task = send_email.delay(email, "Hello")
    return {"task_id": task.id}

@app.get("/result/{task_id}")
def result(task_id: str):
    # 後でtask_idで結果を取りに行く
    from tasks import app as celery_app
    task = celery_app.AsyncResult(task_id)
    return {"status": task.state, "result": task.result}
```

### 6-4. 起動

```bash
# Worker(裏方)
celery -A tasks worker --loglevel=info

# Beat(定期実行用、cron みたいな)
celery -A tasks beat --loglevel=info

# FastAPI
uvicorn main:app
```

→ **メール送信・PDF生成・スクレイピング**を裏で動かす。

> **なぜ別プロセス?**
> Web API は「3秒以内に応答」が鉄則(=客を待たせない)。
> 重い処理は裏に投げ、API はチケット番号だけ返す。

## 7. Redis - キャッシュ・キュー・PubSub(30分)

> **Redis とは?**
> 超高速の「メモ帳」。データをメモリに置くので**マイクロ秒**で読み書き。
> DB(=ミリ秒)より1000倍速い。

```bash
pip install redis
```

### 7-1. キャッシュ

```python
import redis
import json

r = redis.Redis(host="localhost", port=6379, decode_responses=True)

# 高速キャッシュ(1時間で自動削除)
r.set("user:1", json.dumps({"name": "太郎"}), ex=3600)
data = json.loads(r.get("user:1") or "{}")
```

> **なぜキャッシュ?**
> 同じデータを何度もDBから取ると遅い。
> 1回取って Redis に置いておけば、次は瞬時。

### 7-2. キュー(LPUSH / BRPOP)

```python
# Producer(タスク投入)
r.lpush("tasks", "task1")
r.lpush("tasks", "task2")

# Consumer(別プロセス、タスク処理)
while True:
    _, task = r.brpop("tasks", timeout=0)  # キューが空ならブロック
    print(f"Processing: {task}")
```

### 7-3. PubSub(リアルタイム通知)

```python
# Publisher(放送局)
r.publish("notifications", json.dumps({"event": "order_created"}))

# Subscriber(視聴者、別プロセス)
sub = r.pubsub()
sub.subscribe("notifications")
for msg in sub.listen():
    if msg["type"] == "message":
        print(msg["data"])
```

→ チャットアプリ、リアルタイム通知に。

> **PubSub のイメージ:**
> ラジオ放送と同じ。Publisher が放送、Subscriber が聞く。
> Subscriber は何人でもOK。

## 8. レート制限の実装(15分)

> **なぜ?** 1人のユーザーが1秒1000リクエスト送ってきたらサーバー死ぬ。
> 「1分60回まで」と制限する。

```python
# Token Bucket(Redis使用)
import time

def is_allowed(user_id: str, max_per_minute: int = 60):
    key = f"rate:{user_id}"
    current = r.incr(key)  # アトミックに+1
    if current == 1:
        r.expire(key, 60)  # 60秒で自動消滅
    return current <= max_per_minute

if is_allowed("user_123"):
    # API実行
    pass
else:
    # 429 Too Many Requests を返す
    pass
```

## 9. WebSocket - リアルタイム通信(20分)

> **WebSocket とは?**
> 普通のHTTP=「リクエスト→レスポンス」で1往復したら切断。
> WebSocket=「**ずっと繋ぎっぱなし**」、双方向にメッセージ送り合える。

```python
# FastAPI
from fastapi import FastAPI, WebSocket

app = FastAPI()

@app.websocket("/ws")
async def websocket_endpoint(ws: WebSocket):
    await ws.accept()  # 接続受け入れ
    try:
        while True:
            data = await ws.receive_text()  # クライアントから受信
            await ws.send_text(f"echo: {data}")  # 送り返す
    except Exception:
        pass
```

→ チャットボット、ライブ更新ダッシュボード。

## 11. よくある間違い・エラー(15分)

| 状況 | 間違いの原因 | 直し方 |
|---|---|---|
| `async def` 内で時間.sleep() | 普通のsleep使った | `await asyncio.sleep()` を使う |
| asyncio で速くならない | `await`忘れ・直列実行 | `asyncio.gather()` で並行に |
| 大量並行で 429 / BAN | 同時接続多すぎ | `Semaphore` で制限 |
| `await` 忘れ | コルーチンが走らない | 戻り値見て `<coroutine>` ならawait忘れ |
| ProcessPool が遅い | I/O処理に使った | I/O は asyncio、CPU は ProcessPool |
| Celery worker 動かない | 起動忘れ | `celery -A tasks worker` |
| Redis 接続エラー | サーバー未起動 | `brew services start redis` |
| WebSocket 切断ループ | 例外処理なし | `try/except` で守る |
| GIL 知らずスレッド大量 | 思い込み | CPU処理は ProcessPool 必須 |
