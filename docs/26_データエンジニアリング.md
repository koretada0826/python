### なぜ大事?

### イメージ
- **データエンジニアリング**=「**水道工事**」
  - 川(=API・CSV)から水(=データ)を取る
  - 浄水場(=Transform)で綺麗にする
  - 蛇口(=DWH)から各家庭(=分析・AI)に届ける
- **ETL**=「**取り出して(E)、整えて(T)、入れる(L)**」3ステップ作業
- **Airflow**=「**工場の作業ロボット**」(毎朝6時に自動で水道作業)
- **DAG**=「**作業の順番図**」(水を取る → 浄水 → 配水)
- **dbt**=「**SQLでデータを変換するレシピ集**」
- **DWH**=「**データの大型タンク**」(BigQuery・Snowflake など)
- **Parquet**=「**圧縮されたデータ箱**」(CSVより100倍小さく速い)
- **Polars**=「**pandas のスポーツカー版**」

## 1. ETL とは(15分)

### 1-1. 3ステップ

```
Extract(抽出) → Transform(変換) → Load(投入)
```

例:
- **E**(取り出す):CRMからAPIで顧客データ取得
- **T**(整える):住所を統一フォーマットに変換、重複削除
- **L**(投入):DWH(BigQuery)に投入

> **イメージ:**
> - E = 山から木を切ってくる
> - T = 製材所で板に加工
> - L = 家具屋に届ける

### 1-2. ETL vs ELT

最近は **ELT**(先にLoadしてからDWH内でTransform)が主流。

| 方式 | 順番 | 特徴 |
|---|---|---|
| ETL | 取り出し → 整え → 投入 | 古典的、整えてから入れる |
| ELT | 取り出し → 投入 → DWH内で整え | DWH のパワーを活かす(=今主流) |

> **なぜELT主流?**
> BigQuery などのDWHが超強力になり、「DWHの中で SQL で整える」方が速い。
> Tに使うサーバーが要らない=コスト削減。

## 2. Airflow - ワークフロー管理(60分)

「**毎朝6時にCSV取得 → 加工 → メール送信**」みたいなパイプラインを管理。

> **イメージ:** 工場の生産ライン管理者。
> 「Aの作業終わったらBに渡す」「Bが失敗したら通知」を自動でやる。

### 2-1. インストール(Docker推奨)

```bash

curl -LfO 'https://airflow.apache.org/docs/apache-airflow/stable/docker-compose.yaml'
docker compose up
```

→ http://localhost:8080(admin/admin)

### 2-2. DAG(有向非巡回グラフ)の書き方

> **DAG とは?**
> Directed Acyclic Graph = 一方向の流れ図。
> 「A → B → C」のように**逆戻りしない作業の順番**を表す。

`dags/etl_sales.py`:

```python
from datetime import datetime
from airflow import DAG
from airflow.operators.python import PythonOperator

def extract(**ctx):
    # データを外から取ってくる
    import requests
    data = requests.get("https://api.example.com/sales").json()
    # XCom = タスク間の小さなデータ受け渡し袋
    ctx["ti"].xcom_push(key="raw", value=data)

def transform(**ctx):
    # 受け取って整える
    raw = ctx["ti"].xcom_pull(key="raw")
    cleaned = [r for r in raw if r["amount"] > 0]  # 0以下を除外
    ctx["ti"].xcom_push(key="clean", value=cleaned)

def load(**ctx):
    # DWHに保存
    clean = ctx["ti"].xcom_pull(key="clean")
    print(f"Loaded {len(clean)} rows")

with DAG(
    "sales_etl",
    start_date=datetime(2026, 1, 1),
    schedule_interval="0 6 * * *",  # 毎朝6時(cron形式)
    catchup=False,                  # 過去分は遡らない
    tags=["etl"],
) as dag:
    e = PythonOperator(task_id="extract", python_callable=extract)
    t = PythonOperator(task_id="transform", python_callable=transform)
    l = PythonOperator(task_id="load", python_callable=load)

    e >> t >> l  # 実行順:e → t → l
```

→ **毎朝自動実行・失敗通知・再実行**が標準装備。

> **なぜ Airflow?**
> 単に cron で組むと、失敗管理・再実行・履歴が地獄。
> Airflow は GUI で全部見える=本番運用に強い。

## 3. Prefect(モダンな代替)(30分)

Airflow より**書きやすい**新興ツール。

```bash
pip install prefect
```

```python
from prefect import flow, task
import requests

@task(retries=3, retry_delay_seconds=10)  # 失敗時3回リトライ
def extract():
    return requests.get("https://api.example.com/sales").json()

@task
def transform(data):
    return [r for r in data if r["amount"] > 0]

@task
def load(data):
    print(f"Loaded {len(data)} rows")

@flow
def etl_pipeline():
    raw = extract()
    cleaned = transform(raw)
    load(cleaned)

if __name__ == "__main__":
    # 毎朝6時に実行
    etl_pipeline.serve(name="sales-etl", cron="0 6 * * *")
```

→ **Pythonコードそのまま**でワークフロー化。

> **Airflow vs Prefect:**
> - Airflow = 業界標準、大規模、設定多い
> - Prefect = モダン、書きやすい、新規案件向け

## 4. dbt - SQLでデータモデリング(40分)

「**SQLで再利用可能なデータモデルを書く**」。データアナリストの主戦場。

> **イメージ:** SQLを「使い捨て」ではなく「**部品として再利用可能**」にする道具。
> 月別売上を計算するSQL → ファイル化 → 他のSQLから呼び出せる。

### 4-1. インストール

```bash
pip install dbt-postgres dbt-bigquery
```

### 4-2. モデル定義 `models/orders_summary.sql`

```sql
{{ config(materialized='table') }}  -- テーブルとして実体化

SELECT
    DATE_TRUNC('month', created_at) as month,  -- 月で丸める
    user_id,
    COUNT(*) as order_count,                   -- 注文回数
    SUM(amount) as total_amount                -- 合計金額
FROM {{ ref('orders_raw') }}                   -- 別モデルを参照
WHERE status = 'completed'
GROUP BY 1, 2
```

### 4-3. 実行

```bash
dbt run         # モデルをDWHに反映
dbt test        # データ品質チェック(NULL禁止など)
dbt docs serve  # ドキュメント自動生成
```

→ **SQLを再利用・テスト・自動ドキュメント化**できる。

> **なぜ dbt が革命?**
> SQL = 普通は分析者が書いてエクセルにコピペ → 再現性ゼロ。
> dbt = SQL を Git 管理、テスト、ドキュメント化 → ソフトウェアエンジニアリング化。

## 7. データレイク(S3 + Parquet)(20分)

> **データレイク**=「**雑多なデータを生のまま貯める湖**」。
> DWHに入れる前の中間置き場。

```python
import pandas as pd

df = pd.read_csv("data.csv")

# Parquet形式(列指向、超高速・圧縮)
df.to_parquet("data.parquet")

# S3 に直接保存(s3fs)
df.to_parquet("s3://my-bucket/data.parquet")

# 読み込み
df = pd.read_parquet("s3://my-bucket/data.parquet")
```

→ **TB級データ**は CSV ではなく Parquet。

> **CSV vs Parquet:**
> - CSV = 行ごとに保存(=Excel的)
> - Parquet = 列ごとに保存(=「金額列だけ読む」が爆速)
> - サイズも10分の1程度

## 11. よくある間違い・エラー(10分)

| 状況 | 間違いの原因 | 直し方 |
|---|---|---|
| 全データを毎日Transform | Full再実行コスト爆増 | 増分処理(差分だけ) |
| CSV のまま TB データ | サイズ大、遅い | Parquet に変換 |
| Airflow の XCom に大量データ | XComは小さいデータ用 | 大きいデータはS3経由で渡す |
| dbt のテスト書かない | データ壊れても気づかない | 全モデルにNULL check |
| 1個の巨大DAG | 失敗時に全部止まる | DAG分割、依存関係明示 |
| schedule_interval 間違い | cron書式ミス | crontab.guru で確認 |
| catchup=True でデータ重複 | 過去日分も実行された | catchup=False が安全 |
| BigQuery で `SELECT *` | 全列スキャン=高額 | 必要列だけ指定 |
