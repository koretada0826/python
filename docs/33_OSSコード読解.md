## 💡 これは何?(小学生でもわかる説明)

**世界中の天才プログラマーが書いたコードを、自分で読み解く日**

### なぜ大事?
プロのエンジニアが**8割の時間を「読むこと」に使ってる**って知ってた?
- バグ調査(=既存コードを読む)
- 機能追加(=既存コードに加える)

つまり「**書く力**」より「**読む力**」が圧倒的に重要。
なのに学校でも独学でも、ほとんどの人が「書く練習」しかしない。

### イメージ
- **OSS(オープンソース)**=「**世界トップシェフのレシピ本が無料公開**」されてる
- **コード読み**=「**他店の料理を食べて学ぶ**」プロ料理人の修行
- **PR(Pull Request)**=「**シェフに改善案を提案**」して採用されたら名前が載る

> プロの料理人は、必ず他店の料理を食べに行く。
> 自分が知らない味付け・盛り付け・段取りを学ぶため。
> プログラミングも完全に同じで、**自己流の限界を越えるには他人のコードを読むしかない**。

### Claude Code に何を頼めばいいか
この章で学ぶのは「**読み方**」「**何を読むか**」「**どう貢献するか**」。
コード生成は不要だが、**読みながら Claude に質問**するのは超効果的。

## 2. コード読みの基本フロー(20分)

### Step 1: README + Architecture を読む

「**この OSS は何を解決してるか**」を最初に理解する。
中身を見るのは後回し。

> **なぜ?**
> 目的を知らずに森に入ると、迷う。地図(=README)が先。

### Step 2: 全体構造を把握(`tree` コマンド)

```bash
git clone https://github.com/tiangolo/fastapi
cd fastapi
tree -L 2 -I "__pycache__|*.pyc"
```

→ **どこに何があるか**を地図化。

### Step 3: エントリポイント(=入口)から読む

```bash
# FastAPI なら
cat fastapi/__init__.py
cat fastapi/applications.py  # FastAPI クラスが住んでる
```

> **エントリポイントとは?**
> 「`import fastapi` した時、最初に動くファイル」のこと。
> ほとんどのOSSは `__init__.py` から始まる。

### Step 4: 自分の興味ある機能を**追いかける**

例: `app.get("/")` がどう動くか?
1. `@app.get` の定義を探す → `applications.py`
2. `add_api_route` を呼んでる → `routing.py` へ
3. `APIRoute` クラスを読む

> **大事な原則:** **全部読まなくていい**。気になった1本道だけ追う。
> 全部読もうとすると挫折する。

## 3. 必須ツール(15分)

### 3-1. VS Code のショートカット

| キー | 効果 |
|---|---|
| **F12** | 定義(=元の場所)に飛ぶ |
| **Shift+F12** | どこで使われてるか一覧 |
| **Cmd+P** | ファイル名で検索 |
| **Cmd+Shift+F** | 全プロジェクト全文検索 |
| **Cmd+クリック** | F12と同じ |
| **Ctrl+−**(macOSは⌃−)| 元の場所に戻る |

→ コード読みの**必修ショートカット**。これを覚えるだけで読むスピード3倍。

### 3-2. grep / ripgrep(超高速検索)

```bash
brew install ripgrep
rg "class FastAPI" fastapi/   # FastAPIクラスの定義を探す
rg "def get" --type py        # def get を全Pythonファイルから検索
```

> **なぜripgrep?**
> 普通のgrepより10倍速い。`.gitignore`を自動で尊重してくれる。

### 3-3. Sourcegraph

ブラウザで GitHub のコードを検索/ジャンプできる。
ローカルにcloneしなくてもF12相当の機能が使える。

## 5. 実践:FastAPI を読んでみる(60分)

### 課題:`app.get` の中身を追え

```python
from fastapi import FastAPI
app = FastAPI()

@app.get("/")
def root():
    return {"msg": "hello"}
```

→ この `@app.get` はどこで定義されてる?

#### Step1: クローン&検索

```bash
git clone https://github.com/tiangolo/fastapi
cd fastapi/fastapi
rg "def get\(" applications.py
```

→ `class FastAPI` の中に `def get` がある。

#### Step2: 中身を読む

```python
def get(self, path, ...):
    return self.router.get(path, ...)
```

→ `self.router` を探す。`__init__` で `APIRouter` が入ってる。

#### Step3: routing.py を見る

`routing.py` の `APIRouter.get` を見る
→ `add_api_route` を呼んでる
→ `APIRoute` を作って `self.routes` に append

→ これで「**`@app.get`は最終的に Routeリストへの追加**」と理解できる。

> **収穫:**
> こうやって追いかけることで、「**ルーティングって結局リストなんだ**」という本質が見える。
> ブログを100記事読むより1OSSを追う方が学びが深い。

## 6. PR の出し方(30分)

### 6-1. 6ステップ

1. リポジトリを **Fork**(GitHub右上のボタン)
2. **Issue** で「これ困ってる人いる」を確認、コメントで「やります」と宣言
3. 自分のFork で branch 作成(`git checkout -b fix/typo-readme`)
4. 修正・テスト追加(`pytest` でローカル確認)
5. **PR を出す**(Description にIssue番号 `Fixes #42`)
6. レビュー → 修正 → マージ

### 6-2. 最初の PR は何書く?

**typo修正・docstring追加・テスト追加** が定番。

```python
# Before
def calc(x):
    """calc"""
    return x * 2

# After(PR)
def calc(x: int) -> int:
    """Calculate double of x.

    Args:
        x: Input number

    Returns:
        Double of x
    """
    return x * 2
```

→ こんな小さな改善でも**プロの経歴に書ける**。

> **「自分のレベルじゃ無理」と思うな**
> Linus Torvalds(Linux作者)も、最初のPRはtypo修正だった可能性が高い。
> **「貢献した実績」**こそ価値がある。中身は二の次。

### 6-3. PR を出す時のマナー

| 項目 | 内容 |
|---|---|
| タイトル | 簡潔に。`Fix typo in README` など |
| 説明 | なぜ・何を・どうしたか |
| Issue リンク | `Fixes #42` でクローズ自動化 |
| テスト | 修正に対応するテストを追加 |
| CONTRIBUTING.md | 必ず読む。ルール違反だと却下 |

## 8. よくある間違い・エラー表

| 症状 | 原因 | 対処 |
|---|---|---|
| 全部読もうとして挫折 | スコープが広すぎ | 1機能だけ追う、興味あるところだけ |
| F12 が動かない | プロジェクト解析中、または venv 設定ミス | Python環境をVS Codeに認識させる |
| `git clone` が遅い | 大規模リポジトリ | `--depth 1` で浅くclone |
| PRがマージされない | CONTRIBUTING違反、テスト未追加 | リポジトリのガイドを読む、CIを通す |
| 何から読めばいいか迷う | 入口が分からない | `__init__.py` または `main.py` から |
| コードが難しすぎる | レベルが高い | 一段階下のOSSへ。requests→Flask→FastAPI |
