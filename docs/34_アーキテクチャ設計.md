# Day 34:アーキテクチャ設計 - DDD/CQRS/Clean Architecture

## 💡 これは何?(小学生でもわかる説明)

**「動くコード」を「**5年使えるコード**」に格上げする設計図の描き方を学ぶ日**

### なぜ大事?
- 「動くコード」を書けるエンジニア = **腐るほどいる**(月単価60万)
- 「**5年運用に耐える設計**ができるエンジニア」 = **激レア**(月単価150万円圏)

なぜ差がつくか?
新人が書いたコードは、3ヶ月後にバグ修正しようとすると**全部読み直し**になる。
一方、ちゃんと設計されたコードは、3年後でも10秒で「ここに変更を入れればいい」と分かる。
**この差が、企業にとって年間数千万円の差**になる。

### イメージ
- **設計**=「**家を建てる前の設計図**」
  - いきなりハンマー振るう人は、住みにくい家しか建てられない
  - 設計図がある人は、地震・増築・修繕に強い家を建てる
- **DDD**=「**業務の言葉でコードを書く**」(=現場と通じる言葉でレシピを書く)
- **Clean Architecture**=「**重要なルールを真ん中に守る**」(=金庫を一番奥に置く)
- **CQRS**=「**読む人の窓口と書く人の窓口を分ける**」(=銀行の入金口と出金口は別)

> 家を建てる時、いきなり壁を作らないよね。**設計図**を先に描く。
> ソフトウェアも一緒。**デカい建物ほど設計が命**。

### Claude Code に何を頼めばいいか
ここで覚えるのは「**この設計概念は何のため?**」「**どこに何を置く?**」。
具体的なコードは Claude Code に書かせるが、**指示できるようになる必要がある**。

## 1. レイヤードアーキテクチャ(基本)(20分)

### 1-1. 4層の構造

```
┌─────────────────────┐
│ Presentation(API/UI)│ ← FastAPI のエンドポイント、画面
├─────────────────────┤
│ Application(ユースケース)│ ← 「ユーザー作成する」など業務手順
├─────────────────────┤
│ Domain(ビジネスロジック)│ ← User, Order クラス、業務ルール
├─────────────────────┤
│ Infrastructure(DB/外部API)│ ← SQLAlchemy / Stripe / メール送信
└─────────────────────┘
```

> **イメージ:**
> 会社の組織と一緒。
> - 受付(Presentation)=お客様対応
> - 営業(Application)=注文を受ける
> - 商品企画(Domain)=何を売るか決める
> - 倉庫(Infrastructure)=モノを保管・発送

### 1-2. 鉄則:**下が上を知らない**(依存逆転)

- Domain層は Infrastructure層を**知らない**(=DBの存在すら知らない)
- でも結果的にDBに保存される(=後で繋ぐ)

> **なぜそうする?**
> Domain(=業務ルール)はDBが MySQL でも MongoDB でも変わらないはず。
> Domainを純粋に保つことで、**DBを差し替え可能**にする。

## 2. Clean Architecture(40分)

### 2-1. 玉ねぎモデル

```
┌──────────────────────────────────────┐
│ Frameworks(FastAPI/SQLAlchemy)       │ ← 一番外、変わりやすい
│ ┌──────────────────────────────────┐│
│ │ Interface Adapters(Controllers) ││
│ │ ┌────────────────────────────────┐││
│ │ │ Use Cases(create_user)        │││
│ │ │ ┌─────────────────────────────┐│││
│ │ │ │ Entities(User)              ││││ ← 一番中、変わらない
│ │ │ └─────────────────────────────┘│││
│ │ └────────────────────────────────┘││
│ └──────────────────────────────────┘│
└──────────────────────────────────────┘
```

中心ほど**変わらない**、外側ほど**変わりやすい**。
→ DBやフレームワークを差し替えても**コア(ビジネスロジック)は無傷**。

> **なぜ玉ねぎ?**
> 「重要なものを真ん中に、変わりやすいものを外に置く」のが要点。
> 真ん中(Entity)は10年使う、外側(Framework)は3年で変わる。

### 2-2. 実装例

```python
# domain/user.py — 一番中の純粋な業務概念
@dataclass
class User:
    id: int
    name: str
    email: str

# repositories/user_repository.py(抽象)— 「保存できる」とだけ約束
class UserRepository(ABC):
    @abstractmethod
    def save(self, u: User): ...
    @abstractmethod
    def find_by_id(self, id: int) -> User: ...

# infrastructure/sql_user_repository.py(具象)— 実際のDB実装
class SQLUserRepository(UserRepository):
    def __init__(self, session):
        self.session = session
    def save(self, u):
        ...  # SQLAlchemy で保存
    def find_by_id(self, id):
        ...

# application/create_user.py(ユースケース)— 業務手順
class CreateUser:
    def __init__(self, repo: UserRepository):
        self.repo = repo  # 抽象に依存(差し替え可)
    def execute(self, name, email):
        user = User(id=None, name=name, email=email)
        self.repo.save(user)
        return user

# presentation/api.py(FastAPI)— 一番外、UIに近い
@app.post("/users")
def create(name: str, email: str, db = Depends(get_db)):
    use_case = CreateUser(SQLUserRepository(db))
    return use_case.execute(name, email)
```

> **テスト時のメリット:**
> 本物のDBに繋がず、`InMemoryRepository`(メモリ上の偽DB)に差し替えてテストできる。
> テストが**100倍速くなる**。

## 3. DDD(ドメイン駆動設計)(40分)

### 3-1. 一言で

> **「業務の専門家と同じ言葉で**コードを書く」

例:銀行業務なら「振込」「口座」「残高」をそのままクラス名にする。
業務の専門家が見ても**意味が通じる**コードを目指す。

### 3-2. 主要用語

| 英語 | 意味 | 例 |
|---|---|---|
| **Entity** | IDで区別されるモノ | User(id=1), Order(id=42) |
| **Value Object** | 値で区別される(中身が同じなら同じ) | Money(1000円), Address |
| **Aggregate** | 一塊で整合性を保つ単位 | Order + OrderItems(注文と明細) |
| **Repository** | 永続化(=保存)の抽象 | OrderRepository |
| **Service** | 横断的なロジック | PricingService(複数Entity関わる) |
| **Domain Event** | 業務イベント | OrderPlaced(注文された) |

### 3-3. Value Object の例:Money

```python
@dataclass(frozen=True)  # frozen=不変(値オブジェクトの特徴)
class Money:
    amount: Decimal
    currency: str

    def __add__(self, other):
        if self.currency != other.currency:
            raise ValueError("通貨が違う")  # 円とドルを足すのを防ぐ
        return Money(self.amount + other.amount, self.currency)
```

> **なぜこれが大事?**
> 普通に `int` で金額を持つと、**円とドルを足してしまう**バグが起きる。
> `Money` 型にすることで、**型システムが「混ぜたらダメ」と教えてくれる**。
> 金融系で必須の手法。

### 3-4. ユビキタス言語(=みんな同じ言葉を使うルール)

業務担当者と話す時の言葉と、コード上の名前を**一致**させる。

| ❌ ダメな例 | ✅ 良い例 |
|---|---|
| `def fn1(x, y):` | `def calculate_total_with_tax(price, tax_rate):` |
| `t = User(); t.f()` | `user = User(); user.deactivate()` |
| `data` という変数 | `customer_orders` のように具体的に |

> **メリット:**
> ミーティング中に画面共有して、業務担当者が「ああ、それは違う、〇〇のはず」と指摘できる。
> 認識ズレが激減する。

## 4. CQRS(コマンドとクエリ分離)(30分)

### 4-1. 一言で

> **「書き込み」と「読み取り」を別モデルで作る**

```
[書き込みAPI] → Command → 正規化されたDB(整合性厳守)
[読み取りAPI] ← Query ← 高速な専用ビュー(Redis/ES)
```

> **なぜ分ける?**
> 書き込みは「整合性が大事(銀行残高は絶対正確に)」。
> 読み取りは「速さが大事(検索結果はミリ秒で)」。
> 両立は難しいので、**役割を分けて最適化**。

### 4-2. メリット

- 読み取り専用DBで**爆速**(Redis、Elasticsearch)
- 書き込みは整合性厳守、読み取りは速度優先
- 読み書きを別チームで分担できる

### 4-3. 実装例(超簡略)

```python
# === Command(書き込み)===
class CreateOrderCommand:
    user_id: int
    items: list[OrderItem]

class CreateOrderHandler:
    def handle(self, cmd: CreateOrderCommand):
        # 整合性チェック(在庫、金額)
        # DBに保存(正規化されたPostgres)
        # Domain Event 発火(OrderPlaced)
        ...

# === Query(読み取り)===
class GetOrderHistoryQuery:
    user_id: int

class GetOrderHistoryHandler:
    def handle(self, q: GetOrderHistoryQuery):
        # Read DB (Redis/ES) から高速取得
        # JOIN 不要、検索専用に最適化
        ...
```

## 5. Event Sourcing(20分)

### 5-1. 一言で

> **「現在の状態」じゃなく「**起きた出来事の列**」を保存する**

```
通常: users テーブル → name="太郎"  (今の値だけ)

EventSourcing:
  - UserCreated(name="太郎")
  - UserNameChanged(from="太郎", to="次郎")
  - UserDeactivated()
  → イベント列を再生して現在状態を復元
```

> **イメージ:**
> ノートに「今の残高1万円」とだけ書く vs
> ノートに「3000円入金、2000円出金、1万円入金、...」と全部書く。
> 後者は**過去の任意時点に戻れる**(タイムトラベル)。

### 5-2. メリット

- **完全な監査ログ**(いつ誰が何をしたか全部残る)
- **タイムトラベル**(「2024/1/1時点での残高は?」が即答可能)
- **デバッグ最強**(バグの発生瞬間を再現できる)

→ 金融・医療・大規模ECで必須。

## 6. SOLID + DI(依存注入)(20分)

### 6-1. SOLID とは(略)

| 文字 | 意味 | 一言 |
|---|---|---|
| **S** | 単一責任 | 1クラス1役割 |
| **O** | 開放閉鎖 | 拡張に開、変更に閉 |
| **L** | リスコフ置換 | 子クラスは親と置き換え可 |
| **I** | インターフェース分離 | 必要なメソッドだけ |
| **D** | 依存逆転 | 抽象に依存、具体に依存しない |

### 6-2. DI(依存注入)の例

```python
# NG: 具体に依存(差し替え不可、テスト困難)
class OrderService:
    def __init__(self):
        self.db = PostgresDB()  # ← 具体的なDBに直結
        self.email = SendGrid()  # ← 具体的なメールサービスに直結

# OK: 抽象に依存(差し替え可、テスト容易)
class OrderService:
    def __init__(self, db: Database, email: EmailSender):
        self.db = db          # 注入される
        self.email = email    # 注入される
```

> **DIのメリット:**
> - テスト時に偽物(モック)を注入できる
> - 本番ではPostgres、開発ではSQLite、テストではメモリ
> - **結合度が下がる**=変更に強い

### 6-3. FastAPI の DI

```python
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

@app.post("/orders")
def create(order: OrderIn, db = Depends(get_db)):
    ...
```

`Depends(get_db)` で「**DB接続を注入**」する。
テスト時は `app.dependency_overrides[get_db] = mock_db` で差し替え可能。

## 8. よくある間違い・エラー表

| 症状 | 原因 | 対処 |
|---|---|---|
| Clean Arch なのに循環import | レイヤー間の依存方向違反 | 依存逆転(抽象を真ん中に置く) |
| Domain層からSQLAlchemy import | 設計違反、純粋でない | Repositoryで抽象化、Domainは純Python |
| 全クラスが Service クラス | DDDの濫用、責務不明 | Entity/VO/Service を区別 |
| Money を int で持つ | 通貨ミスのバグ | Value Object 化 |
| テストが本物DB必要 | DI使ってない | 抽象+InMemory実装で差し替え |
| CQRS が複雑すぎ | 小さいシステムで過剰設計 | 単純な業務ならレイヤードで十分 |
