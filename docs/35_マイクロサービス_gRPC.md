# Day 35:マイクロサービス・gRPC・Kafka

## 💡 これは何?(小学生でもわかる説明)

**1つの巨大アプリを、小さなアプリの集合体に分解する技術を学ぶ日**

### なぜ大事?
Netflix・Uber・LINE・メルカリのような大規模サービスは、
全部「**マイクロサービス**」で作られている。

- 1つのデカいアプリ = 「**モノリス**」(monolith=一枚岩)
- 小さいアプリの集合 = 「**マイクロサービス**」

なぜ分けるの?
- 100人で1つのコードベースをいじると**毎日衝突する**
- 一部だけスケールしたい(=決済だけ強化)時、全体スケールしか出来ない
- 1個壊れたら**全停止**

ここを理解できる人は、**月単価120万圏**のエンタープライズ案件に入れる。

### イメージ
- **モノリス**=「**1人で全料理を作る巨大ファミレス**」(混乱・遅い・倒れる)
- **マイクロサービス**=「**寿司部・ラーメン部・パン部の分業**」(専門家、独立で動く)
- **gRPC**=「**部署間の社内便**」(高速・型付き伝票)
- **Kafka**=「**伝言板**」(誰が見てもいい掲示板に貼っておく)
- **Service Mesh (Istio)**=「**社内インフラ部署**」(認証・監視を一括管理)

> 巨大ファミレスのキッチン = 1人で全料理 → **混乱**
> →
> **寿司部・ラーメン部・パン部** に分業 → 各々の専門家、独立で動く
>
> これがマイクロサービス。

### Claude Code に何を頼めばいいか
ここで覚えるのは「**いつ分割すべきか?**」「**どう繋ぐか?**」。
具体的な実装は Claude Code に書かせる。

## 1. モノリス vs マイクロサービス(15分)

### 1-1. 比較表

| 項目 | モノリス | マイクロサービス |
|---|---|---|
| 開発 | 簡単(1リポジトリ) | 複雑(N個のリポジトリ) |
| デプロイ | 全部一緒 | 個別にできる |
| スケール | 全部一緒 | 個別にできる |
| 障害 | 全停止 | 1個だけ落ちる |
| 通信 | メソッド呼び出し | ネット越し(遅い) |
| デバッグ | 1プロセスで完結 | 複数プロセス追跡 |
| 学習コスト | 低 | 高 |

### 1-2. 判断基準

> **5人未満ならモノリス、50人超でマイクロ**が経験則。

| 状況 | おすすめ |
|---|---|
| スタートアップ・少人数 | モノリス(まず動かす) |
| 1チームで管理しきれない規模 | マイクロサービス検討 |
| 部分的に違うスケール要件 | マイクロサービス |
| トラフィック数千万req/日 | マイクロサービス必須 |

> **罠:**
> 「最初からマイクロサービス」は失敗の典型。
> モノリスで作って、必要になった部分だけ分割するのが正解。

## 2. 分割の指針(15分)

### 2-1. Bounded Context(DDD)で分ける

業務の**境界**で分けるのが鉄則:
- ユーザー管理サービス
- 注文管理サービス
- 在庫管理サービス
- 決済サービス
- 通知サービス

→ 各サービスが**独立して進化**できる。

### 2-2. やってはいけない分け方

| ❌ ダメな分け方 | 理由 |
|---|---|
| 技術で分ける(DBサービス、APIサービス) | 必ず一緒に変更が必要になる |
| 細かく分けすぎ(Userサービス、Profileサービス) | 通信オーバーヘッド大 |
| データ共有(共通DB) | 結局密結合、独立進化できない |

> **原則:**
> 「**そのサービスは1チームが朝から夜までいじっても他に影響出ない**」レベルで分ける。

## 3. サービス間通信(40分)

### 3-1. REST(同期)— 普通のHTTP

```python
# サービスA から サービスB を呼ぶ
import httpx

async def get_user(user_id: int):
    async with httpx.AsyncClient() as client:
        r = await client.get(f"http://user-service/users/{user_id}")
        return r.json()
```

> **メリット:** シンプル、デバッグ容易、ブラウザでも叩ける
> **デメリット:** 遅い(JSON+HTTP)、型安全じゃない

### 3-2. gRPC(高速・型付き)

```bash
pip install grpcio grpcio-tools
```

`user.proto`:
```protobuf
syntax = "proto3";
service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}
message GetUserRequest { int32 id = 1; }
message User { int32 id = 1; string name = 2; }
```

```bash
# .proto から Python コード自動生成
python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. user.proto
```

→ **REST の3-5倍速**、型安全。Go/Java とも繋がる。

> **なぜ速い?**
> - JSONじゃなくバイナリ形式(Protocol Buffers)で送る
> - HTTP/2 で多重化
> - 1つの接続で複数リクエストを並列処理

### 3-3. メッセージキュー(非同期)— Kafka

```bash
pip install kafka-python
```

```python
from kafka import KafkaProducer, KafkaConsumer
import json

# === Publisher(送信側)===
p = KafkaProducer(
    bootstrap_servers="localhost:9092",
    value_serializer=lambda v: json.dumps(v).encode()
)
p.send("orders", {"order_id": 1, "amount": 1000})

# === Consumer(受信側、別サービス)===
c = KafkaConsumer(
    "orders",
    bootstrap_servers="localhost:9092",
    value_deserializer=lambda v: json.loads(v.decode())
)
for msg in c:
    print(msg.value)
```

→ **イベント駆動**で疎結合化。

> **使い場面:**
> - 「注文された」イベント → 在庫減らす + メール送る + 集計DB更新
> - 各処理が**独立**して動ける、1つ失敗しても他に影響なし

#### RabbitMQ との違い

| 項目 | Kafka | RabbitMQ |
|---|---|---|
| スループット | 超大規模(秒100万メッセージ) | 中規模 |
| 永続化 | ディスク保存、再生可能 | 配信後に消える |
| 順序保証 | パーティション内で保証 | 基本保証 |
| 用途 | ログ、イベントストリーム | ジョブキュー |

### 3-4. 通信方式の選び方

| 状況 | 選択 |
|---|---|
| ユーザーが結果を待つ(同期) | REST or gRPC |
| 内部・性能重視 | gRPC |
| 1リクエストで他に影響不要 | Kafka |
| 障害時に再処理したい | Kafka |

## 4. Service Mesh(Istio)(15分)

### 4-1. 一言で

> **「サービス間通信の共通機能」を一括管理する仕組み**

- 認証・認可
- ロードバランス
- 監視・トレース
- リトライ・サーキットブレーカー

→ Kubernetes + Istio でマイクロサービス本番運用。

> **なぜ必要?**
> サービスが100個あると、各サービスに認証コードを100回書くハメになる。
> Service Meshを使えば**ネットワーク層**で一括処理→各サービスは業務ロジックに集中できる。

### 4-2. サーキットブレーカーとは

> **「壊れた相手に呼びかけ続けて巻き添えで死なないようにする仕組み**

電気のブレーカーと同じ。「下流サービスがダウン」を検知すると、
**自動で呼び出しを止めて、自サービスを守る**。

## 5. 分散トランザクション(20分)

### 5-1. なぜ難しい?

サービスまたいだ「全部成功 or 全部失敗」が課題。
1つのDBなら ACID で守れるが、**複数DBに分散すると守れない**。

### 5-2. Saga パターン

各サービスがイベントを送り合って**段階的に整合**を保つ。

```
1. 注文作成(Order Service) → OrderCreated イベント
2. 在庫減らす(Inventory Service) → 失敗 → CompensateOrder イベント
3. 注文取消(Order Service)
```

> **イメージ:** リレー走者がバトンを渡し、1人転んだら全員「逆走」して戻る。

### 5-3. Outbox パターン

DBに**イベントテーブル**を持ち、トランザクション内でイベントを保存。
別プロセスがそのテーブルを読んで Kafka に publish。

> **なぜ?**
> 「DB保存成功 → Kafka送信失敗」というズレを防ぐため。

## 6. 観測可能性(Observability)(20分)

### 6-1. 3つの柱

1. **Logs**(構造化ログ)— いつ何が起きたか
2. **Metrics**(数値時系列、Prometheus)— どれくらい速い・遅い
3. **Traces**(リクエストの流れ追跡、Jaeger/OpenTelemetry)— どこで詰まったか

```python
# OpenTelemetry
from opentelemetry import trace
tracer = trace.get_tracer(__name__)

with tracer.start_as_current_span("create_order"):
    with tracer.start_as_current_span("validate"):
        ...
    with tracer.start_as_current_span("save_db"):
        ...
```

→ サービスを跨いだ**1リクエストの足跡**が見える。

### 6-2. なぜ重要?

マイクロサービスは「**どこで何が起きたか分からなくなる**」のが最大の弱点。
観測ツールが無いと**障害対応に1日かかる**。
ちゃんと観測してれば**5分で原因特定**できる。

## 8. よくある間違い・エラー表

| 症状 | 原因 | 対処 |
|---|---|---|
| 全部マイクロサービスで作って大失敗 | 過剰設計 | モノリスから始めて分割 |
| サービス間で共通DB | 密結合、独立進化不可 | 各サービス専用DBへ |
| トランザクションが守れない | 分散DB特有問題 | Saga / Outbox パターン |
| デバッグ地獄 | トレースが無い | OpenTelemetry を最初から入れる |
| ネットワーク遅延で遅い | サービス間通信が多すぎ | バルクAPI、Kafka非同期化 |
| gRPC が動かない | proto コンパイル忘れ | `grpc_tools.protoc` 実行 |
| Kafka メッセージが消える | 設定ミス | `acks=all`、レプリカ確認 |
