## 1. インデックスの仕組み(20分)

### 1-1. B-tree(基本)

```sql
CREATE INDEX idx_user_email ON users(email);
```

→ 内部で**ソート済み木構造**を作る。検索 O(log n)。

> **イメージ:** 国語辞典。「あいうえお順」に並んでて、目当ての言葉を二分探索で見つける。
> 100万件でも**20回**で見つかる(2^20 = 約100万)。

### 1-2. 効くクエリ・効かないクエリ

```sql
✅ WHERE email = 'x@y.z'           -- 完全一致
✅ WHERE email LIKE 'x%'            -- 前方一致(=辞書を「あ」から引く)
✅ ORDER BY email                   -- ソート

❌ WHERE email LIKE '%@y.z'         -- 後方一致(=辞書を逆から引けない)
❌ WHERE LOWER(email) = 'x@y.z'     -- 関数で加工すると無効
❌ WHERE email != 'x@y.z'           -- 否定は使えないことが多い
```

> **重要原則:**
> インデックスは「**変形されてない素のカラム**」にしか効かない。
> `LOWER(email)` で検索したいなら、**最初から小文字で保存**するか、**関数インデックス**を作る。

### 1-3. 複合インデックス(複数カラム)

```sql
CREATE INDEX idx_users ON users(country, created_at);
```

→ **順番が大事**:
- `WHERE country='JP'` は使える ✅
- `WHERE country='JP' AND created_at>'2026-01-01'` は使える ✅
- `WHERE created_at='2026-01-01'` だけは**使えない** ❌

> **覚え方:** 電話帳と同じ。「姓→名」順なら姓で検索できるが、名だけでは検索できない。

### 1-4. インデックスのデメリット

| 問題 | 内容 |
|---|---|
| 容量増 | データの30〜50% 追加で必要 |
| INSERT/UPDATE 遅い | 書き込みのたびインデックス更新 |
| 不要なインデックス | プランナーが迷って遅くなる |

→ **何でもインデックス貼ればいい、ではない**。

## 2. EXPLAIN で実行計画を見る(20分)

### 2-1. 基本

```sql
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'x@y.z';
```

> `EXPLAIN`=「あなたどう探す予定?」
> `EXPLAIN ANALYZE`=「実際どう探したか?」(本当に実行する)

### 2-2. 出力の見方

| 出力 | 意味 | 状態 |
|---|---|---|
| `Index Scan` | インデックス使用 | ✅ 速い |
| `Index Only Scan` | インデックスだけで完結 | ✅✅ 最速 |
| `Bitmap Heap Scan` | 中間(複数条件OR) | △ 普通 |
| `Seq Scan` | 全件走査(=端から端まで) | ❌ 遅い |
| `Nested Loop` | JOINがネストループ | △ データ少ないとOK |
| `Hash Join` | ハッシュテーブルJOIN | ✅ 大量データOK |

→ `Seq Scan` 出てたら**インデックス追加** or **クエリ書き換え**。

### 2-3. 読むときのチェックリスト

```
1. Seq Scan が出てないか?(出てたら問題)
2. cost が大きすぎないか?(数万超えは危険)
3. rows(推定行数)と実際の行数が乖離してないか?
4. 同じテーブルを何度もスキャンしてないか?
```

## 3. N+1問題(よくある罠)(15分)

### 3-1. 問題のコード

```python
# NG: ユーザー1人ごとに別クエリ → 100ユーザーで101クエリ
users = session.query(User).all()
for u in users:
    print(u.orders)  # ← 別クエリ実行!
```

> **何が起きてる?**
> 1. `User` 100件を取る → 1クエリ
> 2. ユーザーごとに `orders` を取る → 100クエリ
> 合計**101クエリ**。これがN+1問題。

### 3-2. 解決:JOIN で一発取得

```python
# OK: 1クエリで JOIN
users = session.query(User).options(joinedload(User.orders)).all()
```

→ **応答時間が10倍速**になる。

### 3-3. 検出方法

| ツール | 用途 |
|---|---|
| `SQLALCHEMY_ECHO=True` | 全クエリログ出力 |
| `nplusone` ライブラリ | 自動検出 |
| Django Debug Toolbar | クエリ数を画面表示 |

> **実務での頻度:**
> N+1問題は**新人コードの90%**に潜んでる。
> 見つけて直すだけで「**できる人**」評価になる。

## 4. トランザクション・分離レベル(20分)

### 4-1. ACID(トランザクションの4つの性質)

| 文字 | 名前 | 意味 |
|---|---|---|
| **A** | Atomicity | 全部成功 or 全部巻戻し(原子性) |
| **C** | Consistency | 整合性保つ |
| **I** | Isolation | 同時実行で干渉しない |
| **D** | Durability | コミット後は永続(電源落ちても残る) |

> **例:銀行振込**
> 「Aから引く + Bに足す」が**両方成功 or 両方失敗**じゃないと、お金が消える/増える。
> これが Atomicity。

### 4-2. 分離レベル(同時実行時の見え方)

| レベル | 同時読書の問題 | 速さ |
|---|---|---|
| Read Uncommitted | ダーティリード(未コミットも見える)NG | 最速 |
| Read Committed | 普通(PostgreSQL default) | 速い |
| Repeatable Read | 同じトランザクション内で同じ読み(MySQL default) | 中 |
| Serializable | 完全に直列実行と同じ・最強だが最遅 | 遅い |

### 4-3. デッドロック対策

```python
# トランザクションは短く保つ
with session.begin():
    u = session.query(User).with_for_update().filter_by(id=1).first()
    u.balance -= 100
    # commit
```

→ `with_for_update()` で **悲観ロック**(=他のトランザクションを待たせる)。

> **デッドロック=お見合い問題**
> A: 「先にX取って、次にY」
> B: 「先にY取って、次にX」
> → お互い待ち合って永遠に進まない。
> **対策:全員同じ順序でロック取る**。

## 5. 接続プール(15分)

### 5-1. なぜ必要?

DB接続は**作るのに30〜100ms**かかる。
リクエストごとに作っていたら、接続オーバーヘッドだけで死ぬ。

→ **接続を使い回す**のが接続プール。

```python
from sqlalchemy import create_engine

engine = create_engine(
    "postgresql://...",
    pool_size=20,         # 通常使用する接続数
    max_overflow=10,      # ピーク時に追加で作れる数
    pool_timeout=30,      # 待ちタイムアウト(秒)
    pool_pre_ping=True,   # 使う前に「生きてる?」チェック
)
```

→ **コネクション枯渇**を防ぐ。SaaSで超大事。

> **設定の目安:**
> - `pool_size` = 通常時のDB接続数
> - `max_overflow` = ピーク時の余裕(普通10〜20)
> - 合計が**DBサーバーの max_connections** を超えないように

## 6. Read レプリカ(20分)

### 6-1. 構成

```
[書込み] → [Master DB]
                ↓ レプリケーション(自動コピー)
[読込み] ← [Replica DB] × N台
```

→ **読み取り負荷を分散**。

> **イメージ:**
> 図書館の貸出窓口は1つだが、読書席は20席。
> 借りる(=書き込み)は1人ずつだが、読む(=読み込み)は同時20人OK。

### 6-2. 実装例

```python
# 書き込み(Master)
master_engine.execute(...)

# 読み取り(Replicaからランダム選択)
replica = random.choice(replica_engines)
replica.execute(...)
```

### 6-3. 注意点

| 注意 | 内容 |
|---|---|
| レプリカ遅延 | 書いて即読むと反映前 |
| 強整合性 | 重要な読み取りはMaster |
| トランザクション内 | 一貫性のためMaster固定 |

## 7. パーティショニング(15分)

### 7-1. 一言で

> **「巨大なテーブルを、見えない壁で分割」する技術**

```sql
-- 大きすぎるテーブルを月別に分割
CREATE TABLE orders_2026_01 PARTITION OF orders
  FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
```

→ **過去データ削除も高速**(テーブル削除1回で済む)。

> **メリット:**
> - 検索が速くなる(対象パーティションだけスキャン)
> - 古いデータの削除が一瞬(`DROP TABLE` で済む)
> - インデックスのサイズも小さくて済む

### 7-2. 分割の種類

| 種類 | 例 |
|---|---|
| Range(範囲) | 月ごと、年ごと |
| List(値リスト) | 国別 |
| Hash(ハッシュ) | ID%4で分散 |

## 9. よくある間違い・エラー表

| 症状 | 原因 | 対処 |
|---|---|---|
| `Seq Scan` でクエリ激遅 | インデックス無い、または効いてない | `CREATE INDEX`、関数加工をやめる |
| クエリ数が異常に多い | N+1問題 | `joinedload` / `select_related` |
| 100万件INSERT が遅い | 1件ずつコミット | バルクインサート、トランザクション内一括 |
| デッドロック多発 | ロック順序バラバラ | 全クエリで同じ順序にする |
| `OperationalError: too many connections` | 接続プール設定ミス | `pool_size` 適切設定 |
| レプリカで古いデータ | レプリケーション遅延 | 重要箇所はMaster読み |
| インデックス貼ったのに遅い | カラム加工してる | 関数インデックス or データ正規化 |
