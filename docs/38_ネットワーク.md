### イメージ
ネットは**手紙の郵便局**みたいなもの:
- **IPアドレス** = 「**住所**」(192.168.1.1 のような番号)
- **ポート** = 「**部屋番号**」(同じ家の中で誰宛か区別)
- **TCP/IP** = 「**配達ルール**」(届かなかったら再配達)
- **HTTP** = 「**手紙の中身フォーマット**」(便箋のレイアウト)
- **DNS** = 「**○○太郎の家を住所に変換**」(電話帳)
- **TLS** = 「**手紙を封筒で密封**」(盗み見防止)
- **ファイアウォール** = 「**家の門番**」(誰を入れるか決める)

> ネット世界のあらゆる通信は、結局この**郵便配達モデル**で説明できる。

## 1. OSI 7層モデル(必修)(15分)

### 1-1. 7つの層

| 層 | 役割 | 例 |
|---|---|---|
| 7 アプリ | アプリ通信 | HTTP, SMTP, FTP |
| 6 表現 | 暗号化 | TLS |
| 5 セッション | 接続管理 | - |
| 4 トランスポート | データ転送 | TCP, UDP |
| 3 ネットワーク | ルーティング | IP |
| 2 データリンク | 隣接通信 | Ethernet, Wi-Fi |
| 1 物理 | 信号 | LANケーブル, 電波 |

> **覚え方:** 上から「ある人、生粋の天才ネタっつう物理だ」(覚えなくていい、答えられればOK)。

→ 「**4層の TCP**」「**3層の IP**」を聞かれたら答えられる。

### 1-2. 階層の意味

> **イメージ:** 引っ越し作業の役割分担
> - 7層(アプリ)= 何を運ぶか決める人
> - 4層(TCP/UDP)= 荷造りして発送する人
> - 3層(IP)= 配送ルートを決める人
> - 1層(物理)= トラックの運転手

各層は**自分の仕事だけ**やる。これでネットワークの設計が綺麗に分離される。

## 2. TCP vs UDP(15分)

### 2-1. 比較表

| 項目 | TCP | UDP |
|---|---|---|
| 信頼性 | ◎(再送あり) | ✗(投げっぱなし) |
| 順序保証 | ◎ | ✗ |
| 速度 | 普通 | 速い |
| 用途 | HTTP/SMTP/SSH | DNS/動画/ゲーム |
| ヘッダー | 重い | 軽い |

### 2-2. イメージ

- **TCP** = 「**書留郵便**」
  - 受け取り確認あり、届かなければ再送
  - 確実だが時間かかる
- **UDP** = 「**ビラ配り**」
  - 投げっぱなし、届かなくても気にしない
  - 速いが信頼性低い

### 2-3. 使い分け

| シーン | 選択 |
|---|---|
| Webアクセス | TCP(HTTPはTCP上で動く) |
| ライブ動画配信 | UDP(1フレーム抜けてもOK) |
| オンラインゲーム | UDP(遅延が致命的) |
| メール | TCP(1文字も抜けたらダメ) |

## 3. HTTP の中身(20分)

### 3-1. リクエストの形

```http
GET /api/users HTTP/1.1
Host: example.com
Authorization: Bearer xxx
Content-Type: application/json

{"name": "太郎"}
```

> **構成:**
> 1行目 = メソッド + パス + バージョン
> 2行目以降 = ヘッダー(Authorization、Content-Type など)
> 空行
> ボディ(JSONなど)

### 3-2. ステータスコード(完全版)

```
1xx Informational(情報)
2xx Success(成功)
   200 OK             普通の成功
   201 Created        作成成功(POST)
   204 No Content     成功・中身なし
3xx Redirect(リダイレクト)
   301 Moved Permanently  恒久移動
   304 Not Modified       キャッシュ使え
4xx Client Error(クライアント側のミス)
   400 Bad Request    リクエスト不正
   401 Unauthorized   認証必要
   403 Forbidden      権限なし
   404 Not Found      存在しない
   422 Unprocessable  バリデーション失敗
   429 Too Many Requests  レート超過
5xx Server Error(サーバー側のミス)
   500 Internal Error 内部エラー
   502 Bad Gateway    上流サーバー不調
   503 Unavailable    過負荷・メンテ
   504 Timeout        上流応答なし
```

### 3-3. HTTP/2, HTTP/3

| バージョン | 特徴 |
|---|---|
| HTTP/1.1 | 1接続=1リクエスト(古い) |
| HTTP/2 | 多重化(1接続で並列) |
| HTTP/3 | **UDP**ベース(QUIC)、モバイルで強い |

→ FastAPI は HTTP/2 対応(uvicorn + http2 オプション)。

> **HTTP/3 がUDP?**
> 「TCPは信頼性高いけど遅延がある。UDPベースで信頼性も自前実装」というアプローチ。
> Googleが推進、モバイル環境で特に有利。

## 4. DNS(20分)

### 4-1. 仕組み

> 「**example.com**」を「**93.184.216.34**」に変換する電話帳。

```bash
# 名前解決
dig example.com
nslookup example.com
```

### 4-2. 各レコード

| レコード | 意味 |
|---|---|
| **A** | IPv4 アドレス(普通の住所) |
| **AAAA** | IPv6 アドレス(新しい住所) |
| **CNAME** | 別名(www.example.com → example.com) |
| **MX** | メールサーバー指定 |
| **TXT** | 任意テキスト(SPF/DKIM/Verification) |
| **NS** | ネームサーバー |
| **SOA** | ゾーン情報 |

### 4-3. Pythonから

```python
import socket
ip = socket.gethostbyname("example.com")
print(ip)  # 93.184.216.34
```

### 4-4. DNS 反映遅延

> **DNS 反映遅延**が原因の障害が多い。`dig` で即確認できる。

新しいDNS設定が世界中に行き渡るには**数分〜48時間**かかる。
TTL(Time To Live)で制御。短くすると速く反映するが、DNSサーバー負荷増。

## 5. TLS / HTTPS(15分)

### 5-1. 仕組み(ハンドシェイク)

1. クライアント:「TLS 開始」(対応暗号方式を提案)
2. サーバー:**証明書**を送る
3. クライアント:証明書検証(信頼できる認証局のサインか)
4. 共通鍵交換(Diffie-Hellman 鍵交換)
5. 以後**暗号通信**

> **イメージ:** スパイ映画の合言葉交換。
> 最初は相手を確認(証明書)、その後は二人だけが知る合言葉(共通鍵)で暗号会話。

### 5-2. 証明書の役割

| 確認すること | 内容 |
|---|---|
| 本物のサーバー? | 認証局がサインしてる |
| 期限切れてない? | 有効期限内 |
| ドメイン一致? | example.com の証明書か |

### 5-3. 証明書取得

```bash
# Let's Encrypt(無料)
certbot --nginx -d example.com
```

### 5-4. Python の HTTPS

```python
import requests
r = requests.get("https://example.com", verify=True)   # 検証あり(本番)
r = requests.get("https://example.com", verify=False)  # 検証なし(開発のみ)
```

> **`verify=False` は本番禁止!**
> 中間者攻撃(MITM)を防げない。開発・テスト時のみ使う。

## 6. ロードバランサ・CDN(15分)

### 6-1. ALB(AWS Application Load Balancer)

```
[ALB] ──┬─ [App1]
        ├─ [App2]
        └─ [App3]
```

> **役割:**
> - リクエストを複数サーバーに分散
> - 死んだサーバーを自動的に外す
> - SSL終端(暗号化処理を肩代わり)

### 6-2. CloudFront(CDN)

世界中の**エッジサーバー**にキャッシュ → ユーザーに近い場所から配信。

> **CDN とは?**
> Content Delivery Network。
> 東京のサーバーに置いた画像を、ニューヨーク・ロンドン・シドニーのキャッシュサーバーに自動配置。
> ユーザーは**最寄りのサーバー**から取得 → 爆速。

→ Streamlit Cloud / Render / Vercel は**自動でCDN**してくれる。

## 7. ファイアウォール・セキュリティグループ(15分)

### 7-1. ネットワーク分離

```
[インターネット]
   ↓ (port 80, 443 だけ許可)
[Web サーバー]
   ↓ (port 5432 だけ Web→DB許可)
[DB サーバー]
```

→ **最小権限**(Principle of Least Privilege)。
**必要最低限のポートだけ開ける**のが鉄則。

### 7-2. AWS Security Group 例

```
Inbound(受信):
  TCP 80    0.0.0.0/0       ← 誰でも HTTP
  TCP 443   0.0.0.0/0       ← 誰でも HTTPS
  TCP 22    XX.XX.XX.XX/32  ← 自分のIPだけ SSH
Outbound(送信):
  ALL  0.0.0.0/0            ← 全部許可
```

> **`0.0.0.0/0` の意味:**
> 「全世界からのアクセスを許可」。
> SSH(22番)を `0.0.0.0/0` で開けると**世界中からハッキング試行**が来る。絶対やらない。

## 8. デバッグツール(15分)

### 8-1. 接続確認

```bash
curl -v https://example.com         # 詳細出力(ハンドシェイク見える)
curl -I example.com                  # ヘッダーだけ
ping example.com                     # 疎通確認
traceroute example.com               # 経路追跡
```

### 8-2. ポート確認

```bash
nc -zv example.com 443               # 特定ポート開いてるか
telnet example.com 80                # 古いやり方
```

### 8-3. 詳細パケット解析

```bash
tcpdump -i eth0 port 443            # 443番のパケット監視
wireshark                            # GUI(可視化)
```

### 8-4. サーバー側のリッスン状況

```bash
ss -tlnp                             # 開いてるポート(新)
netstat -tlnp                        # 開いてるポート(旧)
```

### 8-5. 障害切り分けフロー

| 順番 | 確認 | コマンド |
|---|---|---|
| 1 | DNS解決できる? | `dig example.com` |
| 2 | 疎通できる? | `ping example.com` |
| 3 | ポート開いてる? | `nc -zv example.com 443` |
| 4 | HTTPで応答? | `curl -v https://example.com` |
| 5 | 自サーバーで listen? | `ss -tlnp` |

→ この順番で5分で原因特定できる。

## 10. よくある間違い・エラー表

| 症状 | 原因 | 対処 |
|---|---|---|
| `Connection refused` | サーバーがそのポートで待ち受けてない | `ss -tlnp` で確認、サービス起動 |
| `Connection timed out` | ファイアウォールでブロック | Security Group確認 |
| `Name or service not known` | DNS解決失敗 | `dig` で確認、`/etc/hosts` 確認 |
| `SSL: CERTIFICATE_VERIFY_FAILED` | 証明書期限切れ・自己署名 | 証明書更新、認証局確認 |
| `502 Bad Gateway` | バックエンドアプリ落ちてる | アプリ再起動、ログ確認 |
| `429 Too Many Requests` | レート制限 | リクエスト間隔調整、API契約見直し |
| nginx 起動できない | ポート被り | `ss -tlnp` で誰が80番使ってるか |
