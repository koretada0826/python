## 1. JWT 完全理解(30分)

### 1-1. 構造

```
xxxxx.yyyyy.zzzzz
↑      ↑      ↑
Header Payload Signature
```

3つのパートを `.` で繋いだ文字列。それぞれBase64エンコード。

> **イメージ:** 入場券。
> - Header = 「これは入場券です」(種類)
> - Payload = 「○○さん、A席、有効2時間」(中身)
> - Signature = 「主催者のハンコ」(改ざん検知)

### 1-2. 実装

```python
import jwt
SECRET = "your-secret"

# 発行(=入場券を作る)
token = jwt.encode({
    "sub": "user123",
    "exp": datetime.utcnow() + timedelta(hours=1)  # 1時間有効
}, SECRET, algorithm="HS256")

# 検証(=入場券をチェック)
payload = jwt.decode(token, SECRET, algorithms=["HS256"])
```

### 1-3. よくある脆弱性

| ❌ ダメな実装 | 何が起きる? |
|---|---|
| `alg: none` を許可 | 署名なしで認証通過、誰でもなりすまし |
| HS256 で SECRET が弱い | 総当たりで突破される |
| JWT を **localStorage** に置く | XSSで盗まれる |
| 有効期限が長すぎ | 漏洩時の被害が長期化 |
| Refresh Token なし | UX 悪い、毎回ログイン |

### 1-4. 正しい実装

| ✅ 良い実装 | 理由 |
|---|---|
| **httpOnly Cookie** に置く | JSから読めない、XSS対策 |
| `Secure` 属性 | HTTPSでのみ送信 |
| `SameSite=Strict` | CSRF対策 |
| HS256 + 強いランダムSECRET | 256bit以上 |
| 短い有効期限 + Refresh Token | 漏洩リスク最小化 |

## 2. OAuth2 / OIDC(30分)

### 2-1. 「Google でログイン」の仕組み

```
[ユーザー] →[アプリ] → 「Googleにアクセス権ください」
              ↓
[Google] ←──── ログイン → 同意画面 → アクセストークン発行
              ↓
[アプリ] ← トークンで API 呼び出し → ユーザー情報取得
```

> **メリット:**
> - パスワードを自社で管理しなくていい(漏洩リスクゼロ)
> - Googleの2要素認証など最新セキュリティを使える
> - ユーザーも「アカウント増やさなくていい」で楽

### 2-2. Authlib で実装

```bash
pip install authlib
```

```python
from authlib.integrations.starlette_client import OAuth

oauth = OAuth()
oauth.register(
    name="google",
    client_id=os.getenv("GOOGLE_CLIENT_ID"),
    client_secret=os.getenv("GOOGLE_SECRET"),
    server_metadata_url="https://accounts.google.com/.well-known/openid-configuration",
    client_kwargs={"scope": "openid email profile"}
)
```

→ 自社認証作るより**Google任せ**が安全・楽。

### 2-3. 用語

| 用語 | 意味 |
|---|---|
| OAuth2 | 認可(=○○へのアクセス権を委譲)の仕組み |
| OIDC | OAuth2の上に作られた認証(=本人確認)の仕組み |
| Access Token | API呼び出し用のトークン |
| Refresh Token | Access Tokenを更新するトークン |
| Scope | 何への権限か(email, profile等) |

## 3. 暗号(15分)

### 3-1. ハッシュ(一方向、戻せない)

```python
# ❌ MD5/SHA1: 古い、脆弱(衝突攻撃あり)
# ✅ bcrypt / argon2 / scrypt(計算コスト高い=総当たり困難)
from passlib.hash import bcrypt

hash = bcrypt.hash("password123")
bcrypt.verify("password123", hash)
```

> **なぜハッシュ?**
> パスワードを平文DB保存すると、漏洩時に**全ユーザー終わる**。
> ハッシュ化すれば、漏洩しても元のパスワードに戻せない。

> **なぜbcrypt?**
> 計算をわざと遅くしてある(=ストレッチング)ので、総当たり攻撃に強い。

### 3-2. 対称鍵暗号(双方向、同じ鍵)

```python
from cryptography.fernet import Fernet

key = Fernet.generate_key()
f = Fernet(key)
enc = f.encrypt(b"secret")     # 暗号化
plain = f.decrypt(enc)          # 復号化
```

> **使う場面:** ファイル暗号化、DB内の機密データ暗号化。

### 3-3. 公開鍵暗号(双方向、鍵が2つ)

```bash
ssh-keygen -t ed25519
# 公開鍵で暗号化 → 秘密鍵でしか復号できない
```

→ SSH/SSL/デジタル署名で使われる。

> **イメージ:**
> - 公開鍵 = 「**南京錠**」(誰でも閉められる、配ってOK)
> - 秘密鍵 = 「**南京錠の鍵**」(自分だけ持つ、絶対公開しない)

## 4. CSRF / SSRF / XXE(15分)

### 4-1. CSRF(他サイトから自分のAPIを呼ばれる)

> **攻撃シナリオ:**
> 悪意あるサイトに「`<img src="https://yourbank.com/transfer?to=hacker&amount=10000">`」を仕掛ける。
> 銀行にログイン中のユーザーがそのページを開くと、**勝手に送金される**。

防御:
- **SameSite Cookie**(他サイトから来たリクエストにCookie送らない)
- **CSRF Token**(フォームに毎回ランダム値を埋める)

### 4-2. SSRF(サーバーから内部攻撃)

```python
# ❌ ユーザー入力URLをそのまま fetch
url = request.json["url"]
requests.get(url)  # → http://169.254.169.254 でAWSメタデータ取得攻撃

# ✅ ホワイトリスト
ALLOWED_HOSTS = ["api.example.com"]
```

> **攻撃シナリオ:**
> 「URL指定で画像を取得するAPI」に、`http://169.254.169.254/latest/meta-data/`(AWS内部メタデータURL)を指定 → AWS認証情報が漏洩。

### 4-3. XXE(XML 外部エンティティ)

```python
# 古い XML パーサー → 攻撃可能
import xml.etree.ElementTree as ET  # ❌ デフォルト危険

# ✅ 安全版
import defusedxml.ElementTree as ET
```

> **攻撃:** XML内に `<!ENTITY xxe SYSTEM "file:///etc/passwd">` を仕込み、サーバーのファイルを読み出す。

### 4-4. SQLインジェクション(おまけ)

```python
# ❌ 文字列結合
query = f"SELECT * FROM users WHERE name = '{name}'"
# → name="'; DROP TABLE users; --" で全テーブル消える

# ✅ パラメータ化クエリ
cursor.execute("SELECT * FROM users WHERE name = ?", (name,))
```

## 5. 監査ログ(15分)

### 5-1. 「誰が・いつ・何を」したかを全部記録

```python
import structlog
log = structlog.get_logger()

log.info("user_login", user_id=123, ip="1.2.3.4", user_agent="...")
log.info("admin_action", admin_id=1, target_user=123, action="delete")
```

→ JSON で出力、後から検索しやすい。

### 5-2. なぜ重要?

| 用途 | 内容 |
|---|---|
| 不正検知 | 同じIPから1分100回アクセス→攻撃疑い |
| インシデント対応 | 漏洩時に「誰が何取った?」追跡 |
| 法的要件 | GDPR/個人情報保護法で義務 |
| トラブル対応 | 「私はやってない」と言われた時の証拠 |

### 5-3. 何を記録すべき?

- ログイン/ログアウト
- 重要操作(削除、権限変更、決済)
- 認証失敗
- 設定変更
- データエクスポート

## 9. よくある間違い・エラー表

| 症状 | 原因 | 対処 |
|---|---|---|
| パスワード平文DB保存 | ハッシュ化忘れ | `bcrypt.hash()` で必ずハッシュ化 |
| JWTを localStorage に | XSSで盗まれる | httpOnly Cookie に移行 |
| `verify=False` 本番で使用 | MITM攻撃を防げない | 本番は必ず `verify=True` |
| パラメータ化クエリ忘れ | SQLi 脆弱 | ORM使う、`?` プレースホルダ |
| .env を git commit | APIキー流出 | `.gitignore` 必須、git-secrets |
| デフォルトSECRET | 「your-secret」のまま | 256bit ランダム生成 |
| エラーで内部情報漏れ | スタックトレース表示 | 本番は500だけ返す |
| CORS全開放 `*` | 他サイトから呼ばれ放題 | 信頼ドメインのみ許可 |
