# Day 40:テスト戦略 - ピラミッド・E2E・Property

## 💡 これは何?(小学生でもわかる説明)

**「自動味見ロボット」を仕込んで、安心してコードを書く日**

### なぜ大事?
テストを書かないと:
- 修正するたびに**怖くて手が止まる**
- 動作確認に1機能30分かかる
- 本番で不具合発覚 → 深夜呼び出し

テストを書くと:
- **修正しても安心**(壊れたら自動で教えてくれる)
- 動作確認が**全部自動**で1分
- バグの早期発見

→ **「テスト書ける = リファクタ怖くない = 仕事が速い」=単価UP**

### イメージ
- **テスト**=「**自動味見ロボット**」
  - 新しい料理(=新機能)を出す前に、100通りの味見を自動で
  - 失敗したら「ここが違う!」と即フィードバック
- **テストピラミッド**=「**味見の階層**」
  - 1口の味見(=Unit Test)を大量に
  - 完成料理の味見(=E2E Test)は少なめに
- **モック**=「**代役の食材**」
  - 本物の高級食材(=本番DB)を使わず、代用品で味見

> 新しい料理を作って、**味見しないで客に出す** = 危険。
> コードも一緒。**自動味見(テスト)**で安心して新機能追加できる。

### Claude Code に何を頼めばいいか
ここで覚えるのは「**何をテストするか**」「**どの種類か**」。
具体的なテストコードは Claude Code に書かせる。

## 1. テストピラミッド(15分)

### 1-1. 理想形

```
       /\        E2E(画面操作)10%
      /  \
     /----\      統合テスト 20%
    /      \
   /========\    単体テスト 70%
```

→ **下が多く、上が少ない**のが理想。逆ピラミッドは地獄。

### 1-2. なぜこの形?

| 種類 | 速さ | 信頼性 | 数 |
|---|---|---|---|
| Unit | 超速(数ms) | 局所だけ確認 | 多く(数百〜千) |
| Integration | 中(数秒) | 連携も確認 | 中(数十〜百) |
| E2E | 遅(数秒〜分) | ユーザー視点 | 少なく(数件〜数十) |

> **逆ピラミッド(E2Eだらけ)の地獄:**
> - テストに数十分かかる → 実行頻度減 → バグ通る
> - 不安定(画面ちょっと変わると落ちる)
> - メンテ地獄

## 2. Unit Test(基礎)(20分)

Day 18 でカバー済み。pytest 復習。

```python
def test_add():
    assert add(2, 3) == 5

@pytest.mark.parametrize("a,b,expected", [
    (2, 3, 5),
    (-1, 1, 0),
    (0, 0, 0),
])
def test_add_param(a, b, expected):
    assert add(a, b) == expected
```

> **parametrize の力:**
> 1つのテスト関数で複数パターンを一気にチェック。
> エッジケース(0、負数、大きな数)を漏らさず確認。

### 2-1. AAAパターン(良いテストの書き方)

```python
def test_user_creation():
    # Arrange(準備)
    name = "太郎"

    # Act(実行)
    user = create_user(name)

    # Assert(確認)
    assert user.name == name
    assert user.id is not None
```

## 3. Integration Test(20分)

### 3-1. DB込みのテスト

```python
# DB込みのテスト
@pytest.fixture
def db():
    engine = create_engine("sqlite:///:memory:")  # メモリDBで高速
    Base.metadata.create_all(engine)
    yield SessionLocal(bind=engine)

def test_create_user(db):
    user = create_user(db, "太郎", "x@y.z")
    assert db.query(User).count() == 1
```

> **fixture とは?**
> テスト前後の準備・後始末を自動化する仕組み。
> `yield` の前が準備、後が後始末。

### 3-2. なぜインメモリSQLite?

| 選択肢 | 速さ | 本番との一致 |
|---|---|---|
| 本物PostgreSQL(Docker) | 遅 | ◎ |
| インメモリSQLite | 超速 | △(SQL方言違う) |
| モック | 最速 | ✗(DB挙動再現できない) |

→ 普段はインメモリ、重要箇所は本物Docker。

## 4. E2E Test(Playwright)(30分)

### 4-1. セットアップ

```bash
pip install pytest-playwright
playwright install   # ブラウザバイナリをDL
```

### 4-2. 実装例

```python
def test_login(page):
    page.goto("http://localhost:3000/login")
    page.fill("#email", "x@y.z")
    page.fill("#password", "secret")
    page.click("button[type=submit]")
    page.wait_for_url("**/dashboard")
    assert page.locator(".welcome").is_visible()
```

→ **ブラウザを実際に操作**してテスト。ユーザーの体験を保証。

### 4-3. メリット・デメリット

| メリット | デメリット |
|---|---|
| ユーザー視点で確認 | 遅い(数秒〜分) |
| バグ検知率最高 | 不安定(タイミング問題) |
| デモにも使える | メンテコスト高 |

> **コツ:** クリティカルパス(=ログイン、決済、申込み)だけE2Eで守る。

## 5. Property-Based Testing(20分)

### 5-1. 何が違う?

普通のテスト: 「2+3=5になる」と人が書く
**Property Test**: 「**何でもいいから整数2つで成立する性質**」を書く → ライブラリが自動で何百パターン試す

### 5-2. 実装

```bash
pip install hypothesis
```

```python
from hypothesis import given, strategies as st

@given(st.lists(st.integers()))
def test_sort_idempotent(xs):
    """ソートを2回しても結果は同じ"""
    assert sorted(sorted(xs)) == sorted(xs)

@given(st.text())
def test_encode_decode(s):
    """エンコード→デコードで元に戻る"""
    assert decode(encode(s)) == s
```

→ **何百パターン**を自動生成してくれる。バグを見つける確率高い。

### 5-3. なぜ強力?

人間が書くテストは「**思いついたパターンだけ**」。
Property Test は「**0、空文字、巨大文字列、特殊文字、Unicode**」など、人が忘れがちなケースを自動で試す。

## 6. Mocking 高度(15分)

### 6-1. 外部API呼び出しをモック

```python
from unittest.mock import patch, MagicMock

@patch("app.requests.get")
def test_fetch(mock_get):
    # モックの戻り値を設定
    mock_get.return_value.json.return_value = {"data": [1, 2, 3]}

    result = fetch_users()

    assert result == [1, 2, 3]
    # 正しいURLが呼ばれたか確認
    mock_get.assert_called_once_with("https://api.example.com/users")
```

> **なぜモック?**
> - 本物のAPI呼ぶと**料金がかかる**(課金API)
> - 外部依存だと**ネット切れたら落ちる**
> - 異常系(エラー、タイムアウト)を**意図的に再現**できる

### 6-2. モックの種類

| 種類 | 用途 |
|---|---|
| `Mock` | 最低限の代役 |
| `MagicMock` | マジックメソッド対応 |
| `patch` | 既存オブジェクトを差し替え |
| `AsyncMock` | async関数用 |

## 7. カバレッジ + 変異テスト(15分)

### 7-1. カバレッジ(=どの行が実行された?)

```bash
pytest --cov=. --cov-report=html
```

→ HTMLで「**ここの行はテストされてない**」が可視化される。

### 7-2. 変異テスト(=テストの質を測る)

```bash
pip install mutmut
mutmut run
```

→ コードを意図的に**変える**(`>` を `>=` に等)→ テストが落ちなければ「**テスト不足**」と判定。

> **何がいい?**
> カバレッジ100%でも、**ただ実行しただけ**で assert が無いテストは無意味。
> 変異テストは「ちゃんと検知できるテストか」を測る。

### 7-3. 目標カバレッジ

| カバレッジ | 評価 |
|---|---|
| <50% | テスト不足、危険 |
| 50-70% | 普通 |
| 70-90% | 良い |
| >90% | 優秀(ただし無理は禁物) |

## 8. CI でテスト自動化(15分)

`.github/workflows/test.yml`:
```yaml
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: postgres
        ports: [5432:5432]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements.txt
      - run: pytest --cov=. --cov-fail-under=80   # カバレッジ80%未満で失敗
      - uses: codecov/codecov-action@v4
```

> **CI でテスト自動実行する効果:**
> - PR ごとに自動チェック → バグ混入を防ぐ
> - 「私の手元では動くんですけど...」が無くなる
> - レビュアーの負担が減る

## 10. よくある間違い・エラー表

| 症状 | 原因 | 対処 |
|---|---|---|
| テストが本番DB依存 | フィクスチャ不足 | インメモリSQLiteで分離 |
| 並列実行で落ちる | テストが互いに影響 | 各テストで独立した環境 |
| flaky test(たまに落ちる) | 時間・順序・乱数依存 | 固定値、フリーズタイム使用 |
| カバレッジ高いのにバグ多い | assert がない | 変異テストでチェック |
| モックを使いすぎ | テスト本体がモック設定で埋まる | 本物に近い fake 使う |
| E2E が異常に遅い | 全機能をE2Eで | クリティカルパスだけE2E |
| `ImportError` テスト時 | パス通ってない | `pytest.ini` の rootdir 設定 |
