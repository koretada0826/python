- **Embedding** = 「**意味の地図**」
  - 単語や文章を数字の座標に変換 → 似た意味は近くに集まる
- **ベクトルDB** = 「**意味の地図**を高速で検索する図書館」
  - 「この点に近いやつを上位10件取って」を**0.1秒**で返す

### 比喩でひとこと
普通の検索 = 「**この単語と完全一致する本を探せ**」

後者の方が**人間の感覚に近い**。

## 1. Embedding とは(15分)

### 1-1. ざっくり

文章 → **高次元ベクトル(数字の配列、例: 1536次元)** に変換する技術。
意味が近い文章は、ベクトル空間で**近い座標**に配置される。

```python
from openai import OpenAI
client = OpenAI()

resp = client.embeddings.create(
    model="text-embedding-3-large",
    input="Pythonでデータ分析"
)
vec = resp.data[0].embedding  # 1536個の数字のリスト
```

### 1-2. 「近い」をどう測る?

**コサイン類似度**(角度で測る)が標準。

```python
import numpy as np

def cosine(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

print(cosine(vec_a, vec_b))   # 1.0 に近いほど類似
```

> **比喩:** 1536次元の地図上で、**2つの点が同じ方向を向いてるか**を測る。
> 距離じゃなくて**向き**で測るのがコサイン。

## 3. pgvector(超推奨)(30分)

### 3-1. なぜ pgvector が王道?

- **PostgreSQL 1台**で完結 → 運用が単純(別サービス不要)
- **既存のRDB知識**(SQL・トランザクション)がそのまま使える
- **スケールも十分**(数百万ベクトルまで余裕)

### 3-2. セットアップ

```sql
-- 拡張機能を有効化
CREATE EXTENSION vector;

-- ベクトル列を持つテーブル
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding vector(1536)   -- 1536次元のベクトル
);

-- HNSWインデックス(近似最近傍探索、超高速)
CREATE INDEX ON documents
USING hnsw (embedding vector_cosine_ops);

-- 類似検索(<=> がコサイン距離演算子)
SELECT content
FROM documents
ORDER BY embedding <=> '[0.1, 0.2, ...]'::vector
LIMIT 5;
```

### 3-3. 演算子早見

| 演算子 | 意味 |
|---|---|
| `<=>` | コサイン距離(0=完全一致、2=反対) |
| `<->` | L2距離(ユークリッド) |
| `<#>` | 内積(マイナスをつけたもの) |

→ 通常は **`<=>` を使う**。

## 4. ハイブリッド検索(20分)

### 4-1. なぜハイブリッドが必要?

ベクトル検索だけだと**固有名詞が弱い**(「TOYOTA」と「Toyota」が別扱いに)。
キーワード検索だけだと**意味類似が拾えない**。
→ **両方やって統合**するのがハイブリッド。

### 4-2. 実装

```python
def hybrid_search(query, k=10):
    # 1. ベクトル検索
    vec = embed(query)
    vector_results = db.execute("""
        SELECT id, content,
               1 - (embedding <=> %s::vector) as score
        FROM documents
        ORDER BY embedding <=> %s::vector
        LIMIT %s
    """, (vec, vec, k))
    
    # 2. BM25(全文検索、キーワードベース)
    bm25_results = db.execute("""
        SELECT id, content,
               ts_rank(to_tsvector(content), to_tsquery(%s)) as score
        FROM documents
        WHERE to_tsvector(content) @@ to_tsquery(%s)
        ORDER BY score DESC LIMIT %s
    """, (query, query, k))
    
    # 3. RRF(Reciprocal Rank Fusion)で統合
    return rrf_combine(vector_results, bm25_results)


def rrf_combine(list_a, list_b, k=60):
    """順位を逆数で足し合わせて統合"""
    scores = {}
    for rank, doc in enumerate(list_a, start=1):
        scores[doc.id] = scores.get(doc.id, 0) + 1/(k + rank)
    for rank, doc in enumerate(list_b, start=1):
        scores[doc.id] = scores.get(doc.id, 0) + 1/(k + rank)
    return sorted(scores.items(), key=lambda x: -x[1])
```

→ 精度が**飛躍的に**上がる(検索品質+20-40%が普通)。

## 5. Re-ranking(15分)

### 5-1. なぜ Re-rank?

ベクトル検索は**速いが精度はそこそこ**。
そこで:
1. ベクトルで**Top 100**を高速取得
2. **Cross-Encoder**(より重いが精度高いモデル)で並び替え
3. **Top 10**だけを LLM に渡す

→ **2段階検索**で「速さ」と「精度」を両立。

### 5-2. 実装

```python
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-12-v2")

# 1. ベクトル検索でTop 100候補
candidates = vector_search(query, k=100)

# 2. クエリと各候補のペアを作る
pairs = [(query, doc.content) for doc in candidates]

# 3. Re-ranker でスコアリング
scores = reranker.predict(pairs)

# 4. 上位10件
top10 = sorted(zip(candidates, scores), key=lambda x: -x[1])[:10]
```

> **比喩:** 1次選考(ベクトル)で書類選考、2次選考(Re-rank)で面接。
> **2段階で絞る**から精度が上がる。

## 6. Chunking 戦略(15分)

### 6-1. なぜ Chunking が大事?

文書まるごと1ベクトルにすると、**長すぎて意味がぼやける**。
適切に**分割(chunk)**することで、検索精度が**劇的に変わる**。

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,            # 1チャンク 500文字
    chunk_overlap=50,          # 50文字ぶん前後と重ねる
    separators=["\n\n", "\n", "。", " ", ""]  # 段落→改行→句点 で優先的に切る
)
chunks = splitter.split_text(text)
```

### 6-2. 戦略の比較

| 戦略 | チャンクサイズ | 向き |
|---|---|---|
| 細かく(200字) | 短い | FAQ、定義集 |
| 中ぐらい(500字) | 標準 | 一般文書(おすすめ) |
| 長め(1000字) | 長い | コンテキスト重視(契約書) |
| **段落単位** | 可変 | Markdown/ブログ |
| **見出し単位** | 可変 | 構造化された文書 |

> **Tips:** 重なり(overlap)を入れないと**境界の情報が消える**。
> 必ず10-20%程度overlap させる。
